<details>
<summary>1. TCP는 왜 2-way handshake가 아니라 3-way handshake가 필요할까?</summary>

2-way handshake만 하면 클라이언트는 서버가 응답했다는 걸 알 수 있지만, 서버는 클라이언트가 자신의 응답을 제대로 받았는지 확신할 수 없다. TCP는 양방향 통신이기 때문에 양쪽 모두 “상대가 살아 있고, 데이터를 주고받을 준비가 됐다”는 사실을 확인해야 한다.

또 오래된 연결 요청이 네트워크에 남아 있다가 뒤늦게 도착하는 경우도 있을 수 있다. 3-way handshake는 이런 잘못된 연결 생성을 줄이고, 양쪽의 초기 순서 번호를 맞추는 역할도 한다.

정리하면, **3-way handshake는 양쪽이 모두 송신과 수신이 가능한 상태인지 확인하고, 신뢰성 있는 TCP 연결 상태를 만들기 위한 과정**이다.

</details>

<details>
<summary>2. 패킷 손실이 없으면 네트워크 혼잡 문제는 해결된 것일까?</summary>

아니다. 패킷 손실이 없더라도 혼잡 문제는 남을 수 있다. 예를 들어 라우터 버퍼가 매우 크거나 무한하다면 패킷은 버려지지 않고 계속 저장될 수 있다. 하지만 그만큼 큐에서 기다리는 시간이 길어져 지연이 커진다.

즉, 손실은 없지만 응답이 너무 늦어지는 문제가 생긴다. 실제 네트워크에서는 버퍼가 유한하기 때문에 버퍼가 가득 차면 패킷이 버려지고, TCP는 이를 다시 보내면서 네트워크 부하를 더 키울 수도 있다.

정리하면, **혼잡의 비용은 패킷 손실뿐 아니라 큐잉 지연, 불필요한 재전송, 링크 자원 낭비까지 포함한다. 그래서 혼잡 제어는 단순히 패킷을 안 버리게 하는 것이 아니라, 네트워크에 들어가는 데이터 양을 조절하는 문제**다.

</details>

<details>
<summary>3. QUIC은 UDP 위에서 TCP 같은 기능을 직접 구현하는데, 왜 굳이 TCP를 고치지 않고 이런 방식을 택했을까?</summary>

TCP는 운영체제 커널과 네트워크 장비에 깊게 들어가 있어서 바꾸기 어렵다. TCP 자체를 개선하려면 사용자 운영체제, 서버 운영체제, 방화벽, 공유기, 중간 장비 등이 모두 그 변화를 지원해야 한다. 그래서 변화가 느리고 호환성 문제도 크다.

반면 UDP는 단순하고 이미 널리 지원된다. QUIC은 UDP를 기본 운반 수단으로 사용하고, 그 위에 신뢰적 전송, 재전송, 혼잡 제어, 연결 관리, 암호화 같은 기능을 애플리케이션 계층에서 직접 구현한다.

이렇게 하면 브라우저나 서버 프로그램 업데이트만으로 빠르게 개선할 수 있고, TCP 연결 후 TLS 연결을 따로 맺는 과정도 줄일 수 있다. 또 여러 스트림을 독립적으로 관리해서 하나의 스트림이 막혀도 다른 스트림은 계속 진행할 수 있다.

정리하면, **QUIC은 TCP의 좋은 기능을 버린 것이 아니라, TCP를 직접 바꾸기 어려운 현실 때문에 UDP 위에서 더 빠르고 유연하게 다시 구현한 방식**이다.

</details>

<details>
<summary>4. TCP slow start vs exponential backoff</summary>

Slow start는 TCP 혼잡 제어에서 cwnd를 키우는 방식.
처음에는 작게 시작하지만 ACK가 잘 오면 cwnd를 빠르게 늘려, 보통 RTT마다 대략 2배씩 커지니까 실제 증가 형태는 exponential growth에 가깝다.

반면 exponential backoff는 실패했을 때 재시도 간격을 점점 늘리는 방식.
예를 들어 타임아웃이 반복되면 바로바로 다시 보내지 않고 기다리는 시간을 2배씩 늘린다.

slow start = 성공하면 전송량을 지수적으로 늘림
exponential backoff = 실패하면 재시도 대기 시간을 지수적으로 늘림

</details>
