## 3.5 연결지향형 트랜스포트:TCP

### 1. 통신의 시작: 3-Way Handshake (연결 설정)

TCP는 데이터를 보내기 전에 반드시 양방향 호스트가 서로 인사하며 논리적인 연결을 맺는 과정을 거칩니다. 이를 **3-way handshake**라고 합니다.

- **왜 먼저 인사할까요?** 서로의 송수신 버퍼(메모리)를 할당하고, 데이터 조립에 기준이 될 '초기 순서 번호(Initial Sequence Number)'를 동기화하기 위해서입니다.
- **과정:**
    1. **클라이언트 ➡️ 서버 (SYN):** "안녕, 통신 시작하자. 내 초기 순서 번호는 X야."
    2. **서버 ➡️ 클라이언트 (SYN-ACK):** "그래 좋아. 네 번호 X 잘 받았고(ACK X+1), 내 초기 순서 번호는 Y야."
    3. **클라이언트 ➡️ 서버 (ACK):** "확인했어. 네 번호 Y 잘 받았어(ACK Y+1). 이제 데이터 보낼게!"

### 2. 누적 응답 (Cumulative ACK)과 순서 번호

TCP의 세그먼트 헤더는 UDP(8바이트)보다 훨씬 무거운 **20바이트**입니다. 이 공간의 대부분을 차지하는 핵심이 바로 순서 번호(Seq)와 확인 응답 번호(ACK)입니다.

- **순서 번호 (Seq):** 전송하는 데이터의 '바이트(Byte) 단위 연속된 번호'입니다. 패킷 1, 2, 3이 아니라 바이트 스트림을 기준으로 번호를 매깁니다.
- **누적 ACK:** TCP의 ACK 번호는 "나 N번까지 다 잘 받았어"가 아니라, "내가 다음에 받아야 할 번호는 N번이야(수신 기대 번호)"라는 의미를 가집니다.
    - *예시:* 서버가 0~99바이트를 잘 받았다면, 클라이언트에게 `ACK 100`을 보냅니다.

### 3. TCP의 핵심 최적화: 빠른 재전송 (Fast Retransmit)

3.4절에서 배운 타임아웃 메커니즘은 치명적인 단점이 있습니다. 인터넷 환경에서는 타임아웃 주기를 꽤 길게 잡아야 하는데, 패킷이 하나 유실되면 그 긴 시간을 하염없이 기다려야 한다는 것입니다. TCP는 이를 극복하기 위해 **빠른 재전송**이라는 기법을 씁니다.

> **💡 빠른 재전송의 원리**
> 
> 1. 송신자가 패킷 1, 2, 3, 4, 5를 연속으로 보냅니다.
> 2. 패킷 2가 유실되었습니다. 하지만 3, 4, 5는 수신자에게 무사히 도착합니다.
> 3. 수신자는 3, 4, 5를 받을 때마다 순서가 비어있음을 인지하고, 송신자에게 **"나는 아직 2번이 필요해!"라며 `ACK 2`를 반복해서(중복으로) 보냅니다.**
> 4. 송신자가 동일한 **중복 ACK를 3번** 연속으로 받으면, 타임아웃 타이머가 아직 끝나지 않았더라도 "아, 2번이 확실히 날아갔구나!"라고 판단하고 **즉시 2번 패킷을 재전송**합니다.

### 4. 수신자 보호: 흐름 제어 (Flow Control)

송신자가 데이터를 너무 빠르게 쏟아내어 수신자의 메모리 버퍼가 넘쳐(Overflow) 패킷이 버려지는 것을 막는 기능입니다.

- **수신 윈도우 (Receive Window, rwnd):** 수신자는 자신에게 보내는 ACK 패킷의 헤더에 "나 지금 버퍼에 여유 공간이 이만큼(rwnd) 남았어"라는 정보를 실어 보냅니다.
- 송신자는 이 `rwnd` 값을 보고, 절대 그 여유 공간을 초과하는 양의 데이터를 한 번에 보내지 않도록 스스로 송신 속도를 조절합니다.

### 1. 피기배킹 (Piggybacking)

TCP는 효율성을 극도로 추구합니다. 클라이언트와 서버가 서로 데이터를 주고받는 상황(예: Telnet이나 채팅)을 가정해 봅시다.

- 서버가 클라이언트로부터 데이터를 받고 "잘 받았어(ACK)"라는 응답만 달랑 보내면 헤더 오버헤드(20바이트)가 낭비됩니다.
- 그래서 서버는 즉시 ACK 패킷을 보내지 않고 잠시 기다렸다가, **자신이 클라이언트에게 보낼 새로운 데이터가 생겼을 때 그 데이터 패킷의 헤더에 ACK 번호를 '업혀서(Piggyback)' 함께 보냅니다.** 이를 통해 네트워크 트래픽을 크게 절약합니다.

### 2. TCP 상태 전이도 (State Transition Diagram)

- 서버가 클라이언트를 기다리는 `LISTEN` 상태
- 연결 요청을 보낸 `SYN_SENT` 상태
- 통신이 한창인 `ESTABLISHED` 상태
- **TIME_WAIT 상태:** 4-way handshake의 마지막에 클라이언트가 연결을 완전히 끊기 전, 혹시라도 네트워크에 떠돌아다니는 지연된 패킷이나 자신이 보낸 마지막 ACK가 유실되었을 상황을 대비해 **잠시 동안(보통 1~2분) 연결 흔적을 남겨두고 대기하는 매우 중요한 상태**입니다. (서버 개발 시 포트 충돌 문제로 자주 마주하게 됩니다.)

### 3. SYN 플러딩(Flooding) 공격과 SYN 쿠키 방어 (보안)

- **SYN 플러딩 (DoS 공격):** 해커가 수만 개의 가짜 IP로 서버에 연결 요청(SYN)만 무더기로 보냅니다. 서버는 규칙대로 메모리(버퍼)를 할당하고 응답(SYN-ACK)을 보낸 뒤 하염없이 마지막 ACK를 기다리게 되는데, 결국 서버의 메모리가 고갈되어 정상적인 사용자를 받지 못하게 됩니다.
- **SYN 쿠키 (방어):** 이를 막기 위해 현대의 TCP는 SYN 요청을 받았을 때 당장 메모리를 할당하지 않고, 특별한 암호화 해시 함수를 돌려 만든 기출문제(Cookie)를 순서 번호로 만들어 보냅니다. 클라이언트가 이 쿠키를 포함해 정상적인 마지막 ACK를 보내야만 그때 비로소 메모리를 할당하는 스마트한 방어 기법입니다.

## 3.6 혼잡 제어의 원리

### 1. 혼잡(Congestion)이란 무엇인가?

쉽게 말해 "너무 많은 컴퓨터가, 네트워크가 처리할 수 있는 속도보다 너무 빠르게, 너무 많은 데이터를 전송하려고 시도할 때 발생하는 현상"입니다.
명절 연휴에 모든 차들이 톨게이트로 쏟아져 나와 고속도로 전체가 거대한 주차장으로 변하는 것과 완벽히 똑같은 원리입니다.

### 2. 혼잡으로 인해 발생하는 3가지 치명적인 비용 (Costs of Congestion)

교재에서는 네트워크가 막힐 때 발생하는 피해를 3가지 단계로 설명합니다.

- **비용 1: 엄청난 큐잉 지연 (Large Queuing Delays)**
    - 라우터의 처리 용량(대역폭) 한계에 근접할수록, 패킷들은 라우터 내부의 큐(대기열)에 쌓이게 됩니다. 패킷이 목적지에 도달하는 시간이 기하급수적으로 늘어납니다.
- **비용 2: 불필요한 재전송의 늪 (Retransmission Overhead)**
    - 큐가 꽉 차면 라우터는 어쩔 수 없이 새로 들어오는 패킷을 버립니다(Drop).
    - 그러면 송신자는 타임아웃이 터져서 **안 그래도 막히는 도로에 똑같은 패킷을 또 밀어 넣습니다.** (혼잡이 혼잡을 낳는 악순환입니다.)
- **비용 3: 업스트림 대역폭의 낭비 (Wasted Upstream Capacity)**
    - 가장 뼈아픈 비용입니다. 패킷이 목적지 바로 앞의 4번째 라우터에서 큐가 꽉 차서 버려졌다고 가정해 봅시다.
    - 그 패킷을 4번째 라우터까지 힘들게 실어 나른 1, 2, 3번째 라우터의 땀방울(대역폭과 처리 자원)은 전부 헛수고가 됩니다.

### 3. 혼잡에 대응하는 2가지 접근법

그렇다면 이 꽉 막힌 도로를 어떻게 뚫어야 할까요? 교재는 크게 두 가지 설계 철학을 제시합니다.

### A. 종단 간 혼잡 제어 (End-to-End Congestion Control)

- **방식:** 네트워크 중간에 있는 라우터들은 송신자에게 "나 지금 막힌다!"라고 아무런 말도 해주지 않습니다. (인터넷의 기본 철학인 '단순한 네트워크 코어' 유지)
- **원리:** 송신자가 **눈치껏** 알아채야 합니다. 패킷을 보냈는데 타임아웃이 나거나 3번의 중복 ACK가 오면, 송신자는 "아하, 중간에 어느 라우터 큐가 꽉 차서 내 패킷이 버려졌구나? 도로가 막히네!"라고 스스로 유추하고 송신 속도를 대폭 줄입니다.
- **대표 주자:** 우리가 쓰는 **TCP**가 정확히 이 방식을 사용합니다.

### B. 네트워크 지원 혼잡 제어 (Network-assisted Congestion Control)

- **방식:** 라우터가 직접 송신자나 수신자에게 명시적으로 피드백을 줍니다.
- **원리:** 라우터 내부의 큐가 어느 정도 차오르면, 지나가는 패킷의 헤더에 "나 지금 막히기 시작했으니까 속도 좀 줄여!"라는 경고 스티커(ECN 비트 등)를 붙여서 보냅니다. 이를 받은 수신자가 송신자에게 이 사실을 알려주어 속도를 줄이게 합니다.

## 3.7 TCP 혼잡 제어

앞서 3.6절에서 라우터들은 송신자에게 "나 지금 막힌다"라고 친절하게 알려주지 않는다고 배웠습니다. 따라서 TCP는 순전히 **자신이 보내는 패킷의 성공(ACK)과 실패(유실)라는 '눈치'만으로** 전 세계 네트워크의 텅 빈 공간(대역폭)을 스스로 찾아내야 합니다.

이를 위해 TCP는 송신자 내부에 혼잡 윈도우(Congestion Window, cwnd)라는 변수를 도입합니다. 송신자는 수신자가 허락한 공간(`rwnd`)과 자신이 파악한 네트워크의 여유 공간(`cwnd`) 중 **더 작은 값**을 기준으로 데이터를 전송합니다.

TCP가 이 `cwnd` 크기를 조절하는 3가지 핵심 알고리즘은 다음과 같습니다.

### 1. 슬로우 스타트 (Slow Start): 기하급수적 탐색

처음 통신을 시작할 때, 송신자는 네트워크에 여유 공간이 얼마나 있는지 전혀 모릅니다.

- **동작:** 초기 `cwnd`를 1 MSS(Maximum Segment Size)로 아주 작게 시작합니다. 하지만 정상적인 ACK를 받을 때마다 `cwnd`를 **2배씩(1 $\rightarrow$ 2 $\rightarrow$ 4 $\rightarrow$ 8...) 기하급수적으로** 늘립니다.
- **의미:** 이름은 'Slow'지만, 실제로는 대역폭의 한계를 찾기 위해 미친 듯이 속도를 끌어올리는 단계입니다. 이 폭발적인 증가는 사전에 설정된 임계값인 ssthresh (Slow Start Threshold)에 도달할 때까지 계속됩니다.

### 2. 혼잡 회피 (Congestion Avoidance): 선형적 탐색 (AIMD)

`cwnd`가 `ssthresh` 값에 도달하면, TCP는 "아, 이제 슬슬 한계에 다가오고 있구나"라고 판단하고 폭주를 멈춥니다.

- **동작:** 이제부터는 RTT(왕복 시간)마다 `cwnd`를 2배씩 늘리지 않고, **딱 1 MSS씩만 아주 조심스럽게(선형적으로) 늘려갑니다.** (Additive Increase)
- **의미:** 네트워크가 버틸 수 있는 진짜 한계점을 찾기 위해 조금씩 속도를 높여가며 간을 보는 단계입니다.

### 3. 혼잡 발생 시의 대처 (빠른 회복 / 상태 초기화)

속도를 계속 올리다 보면 결국 라우터의 큐가 터져 패킷이 유실되는 '혼잡'이 발생합니다. TCP는 이 유실의 **심각도**에 따라 다르게 대처합니다.

- **상황 A: 3번의 중복 ACK 수신 (경미한 혼잡 - 빠른 회복)**
    - **판단:** "패킷 하나가 빠지긴 했지만, 뒤이은 패킷들은 잘 도착해서 중복 ACK를 보내고 있네? 도로가 꽉 막힌 건 아니고 잠깐 덜컹거린 수준이군."
    - **대처:** 현재 `cwnd`의 절반 값을 새로운 `ssthresh`로 설정합니다. 그리고 `cwnd`를 1로 떨어뜨리지 않고, 방금 정한 **`ssthresh` 값으로 반토막**만 낸 뒤 다시 '혼잡 회피(1씩 증가)'를 이어갑니다. (Multiplicative Decrease)
- **상황 B: 타임아웃 발생 (심각한 혼잡)**
    - **판단:** "아무런 응답이 안 오네. 라우터가 완전히 뻗어서 패킷이 통째로 증발했구나!"
    - **대처:** 현재 `cwnd`의 절반 값을 새로운 `ssthresh`로 설정하는 것은 똑같지만, **`cwnd`를 가차 없이 1로 초기화**해버리고 다시 '슬로우 스타트(2배씩 증가)'부터 시작합니다.

이러한 "조금씩 1씩 늘리다가(Additive Increase), 문제 생기면 절반으로 뚝 자르는(Multiplicative Decrease)" TCP의 특성 때문에, 혼잡 윈도우 그래프는 유명한 **톱니바퀴(Sawtooth)** 모양을 그리게 됩니다.

## 3.8 트랜스포트 계층 기능의 발전

### 1. 왜 TCP를 버리고 새로운 프로토콜을 만들었을까? (TCP의 3가지 한계)

수십 년간 훌륭하게 작동해 온 TCP이지만, 현대의 초연결, 대용량 웹 환경에서는 다음과 같은 치명적인 한계점들이 드러나기 시작했습니다.

- **한계 1: 무거운 초기 연결 (Handshake Overhead)**
    - 안전한 웹 서핑을 위해 오늘날 거의 모든 통신은 암호화(TLS/SSL)를 사용합니다.
    - TCP로 통신하려면 TCP 자체의 3-Way Handshake(1 RTT 소모)를 거친 후, 암호화 키를 교환하는 TLS Handshake(1~2 RTT 소모)를 또 거쳐야 합니다. 데이터는 보내지도 않았는데 인사하는 데만 시간이 낭비됩니다.
- **한계 2: 헤드 오브 라인 블로킹 (Head-of-Line Blocking, HOLB)**
    - 하나의 TCP 연결 안에서 여러 개의 파일(예: 이미지 1, 2, 3)을 동시에 다운로드한다고 가정해 보겠습니다.
    - TCP는 순서를 목숨처럼 지키는 프로토콜입니다. 만약 맨 앞의 '이미지 1' 패킷이 유실되면, **뒤에 정상적으로 도착한 '이미지 2', '이미지 3'의 패킷들도 이미지 1이 재전송될 때까지 꼼짝없이 수신 버퍼에 갇혀 대기해야 합니다.**
- **한계 3: 운영체제(커널) 종속성**
    - TCP는 윈도우나 리눅스 같은 운영체제의 '커널(심장부)'에 깊숙이 내장되어 있습니다. 새로운 TCP 혼잡 제어 알고리즘을 개발해도, 전 세계 사용자들이 자신의 운영체제를 업데이트하지 않는 이상 적용할 방법이 없습니다.

### 2. 구원자의 등장: QUIC (Quick UDP Internet Connections)

이러한 문제를 해결하기 위해 구글(Google)이 주도하여 만든 프로토콜이 바로 QUIC입니다. 재미있는 점은 이름에서도 알 수 있듯, **QUIC은 TCP를 개조한 것이 아니라 완전히 텅 빈 도화지 같은 'UDP' 위에서 동작하도록 설계되었다는 것**입니다.

UDP는 에러 복구도 안 되고 순서도 보장하지 않지만, 그만큼 가볍고 제약이 없습니다. QUIC은 이 UDP를 기반으로 애플리케이션 단에서 신뢰성과 암호화 기능을 몽땅 새롭게 구현해 냈습니다.

### 3. QUIC의 3가지 핵심 혁신

- **혁신 1: 0-RTT / 1-RTT 초고속 연결 설정**
    - QUIC은 TCP 연결과 TLS 암호화 연결을 하나로 합쳤습니다.
    - 처음 만나는 서버와는 단 한 번의 왕복(1-RTT)만으로 보안 연결까지 끝내버립니다. 게다가 **이전에 한 번 접속했던 서버라면, 인사 과정 아예 없이 곧바로(0-RTT) 데이터를 전송**하기 시작합니다.
- **혁신 2: 독립적인 멀티플렉싱 (HOLB 완벽 해결)**
    - QUIC은 하나의 통신 안에 여러 개의 '독립적인 스트림(Stream)'을 만듭니다.
    - 만약 '이미지 1' 스트림에서 패킷이 유실되어 멈추더라도, '이미지 2'와 '이미지 3' 스트림은 전혀 영향을 받지 않고 목적지 애플리케이션으로 쏙쏙 빠져나갑니다.
- **혁신 3: 유저 스페이스(User-space) 동작**
    - QUIC은 운영체제 커널이 아니라 크롬(Chrome) 같은 웹 브라우저나 애플리케이션 단에 설치됩니다. 따라서 운영체제 업데이트를 기다릴 필요 없이, 브라우저 업데이트만으로 전 세계 통신 프로토콜을 순식간에 진화시킬 수 있습니다.
