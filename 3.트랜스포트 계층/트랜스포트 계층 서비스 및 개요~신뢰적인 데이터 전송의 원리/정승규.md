## 3.1 트랜스포트 계층 서비스 및 개요

### 1. 트랜스포트 계층의 핵심 역할: 논리적 통신

트랜스포트 계층은 각기 다른 호스트에서 실행되는 애플리케이션 프로세스 간의 논리적 통신(Logical Communication)을 제공합니다.

- **논리적 통신이란?** 프로세스들이 서로 물리적으로 어떻게 연결되어 있는지(중간에 라우터가 몇 개인지, 광케이블인지 무선인지) 상관없이, 마치 직접 연결된 방에서 대화를 나누는 것처럼 느끼게 해주는 것을 의미합니다.
- **구현 위치:** 트랜스포트 계층 프로토콜은 오직 종단 시스템(End System, 호스트)에서만 실행됩니다. 네트워크 중간에 있는 라우터들은 트랜스포트 계층의 패킷(세그먼트) 내부를 들여다보거나 제어하지 않으며, 오직 네트워크 계층(IP) 수준에서만 작동합니다.

### 2. 트랜스포트 계층(Transport) vs 네트워크 계층(Network)

이 둘의 차이를 이해하는 것이 3.1절의 가장 중요한 이정표입니다. 교재에서는 이를 이해하기 위해 아주 유명한 '우편물 배달 비유'를 사용합니다.

### 🏠 우편물 배달 비유

- **상황:** 미국의 어느 한 주택에 사는 12명의 아이가 영국의 어느 한 주택에 사는 12명의 아이와 서로 편지를 주고받습니다.
- **집 안의 아이들** = 애플리케이션 프로세스 (Process)
- **집 안의 편지 대장(맏이)** = 트랜스포트 계층 프로토콜 (TCP/UDP)
    - 각 방의 아이들에게 편지를 모아서 봉투에 넣고 우체통에 전달하거나, 도착한 우편물 가방을 열어 각 아이들에게 나누어 주는 역할을 합니다.
- **국가 우편 서비스(우체부와 우체국)** = 네트워크 계층 프로토콜 (IP)
    - 이 집 문앞에서 저 집 문앞까지 우편 가방을 안전하게(?) 실어 나르는 역할을 합니다.

### 짚고 넘어갈 핵심 차이점

- **네트워크 계층:** 호스트와 호스트 간(Host-to-Host)의 전달을 책임집니다. (어느 컴퓨터에서 어느 컴퓨터로 보낼 것인가?)
- **트랜스포트 계층:** 프로세스와 프로세스 간(Process-to-Process)의 전달을 책임집니다. (그 컴퓨터 안의 어떤 프로그램에게 줄 것인가?)

> **⚠️ 하위 계층 한계의 상속과 극복**
트랜스포트 계층은 네트워크 계층이 제공하는 서비스의 한계 안에서 움직입니다. 하위 네트워크 계층(IP)이 대역폭 보장이나 지연 시간 보장을 못 하기 때문에, TCP/UDP 역시 **속도나 지연 시간을 보장하는 서비스는 제공할 수 없습니다.** > 하지만 하위 계층이 패킷을 잃어버리거나 순서를 뒤바꾸더라도(IP의 비신뢰성), 트랜스포트 계층(TCP)은 자체적인 메커니즘을 통해 상위 애플리케이션에게 **'신뢰성 있는 데이터 전송'을 재창조하여 제공**할 수 있습니다.
> 

### 3. 인터넷 트랜스포트 계층의 두 축: TCP vs UDP

인터넷 환경에서 트랜스포트 계층은 두 가지 상반된 프로토콜을 서비스 모델로 제공합니다. 애플리케이션 개발자는 자신의 프로그램 특성에 맞는 프로토콜을 선택해야 합니다.

### A. UDP (User Datagram Protocol)

- **특징:** 비연결형(Connectionless), 비신뢰성(Unreliable) 서비스.
- **하는 일:** 트랜스포트 계층이 해야 할 최소한의 일만 합니다. 호스트 간 전송된 패킷을 포트 번호를 보고 알맞은 프로세스에 넣어주는 **다중화/역다중화(Mux/Demux)** 기능과 간단한 오류 검사(Checksum)가 전부입니다.
- **장점:** 확인 응답이나 연결 설정 과정이 없어 속도가 매우 빠르고 오버헤드가 적습니다.

### B. TCP (Transmission Control Protocol)

- **특징:** 연결지향형(Connection-oriented), 신뢰성 있는 전송(Reliable) 서비스.
- **하는 일:** 패킷 손실 시 재전송, 순서 뒤바뀜 교정, 데이터 흐름 제어 및 네트워크 혼잡 제어까지 수행하는 종합 패키지 서비스입니다.
- **장점:** 데이터가 절대 깨지거나 유실되지 않고 순서대로 도착함을 보장합니다.

## 3.2 다중화와 역다중화

### 1. 다중화(Mux)와 역다중화(Demux)의 기본 정의

컴퓨터 안에는 웹 브라우저, 카카오톡, 게임 등 수많은 네트워크 프로세스가 동시에 돌고 있습니다. 하지만 물리적인 랜카드(네트워크 인터페이스)는 대개 하나뿐입니다. 이 병목을 해결하는 것이 두 개념입니다.

- **역다중화 (Demux, 수신 측):** 트랜스포트 계층이 하위 네트워크 계층(IP)으로부터 세그먼트를 받아, 헤더 정보를 확인한 뒤 **정확한 애플리케이션 소켓(Socket)으로 데이터를 전달하는 과정**입니다.
- **다중화 (Mux, 송신 측):** 여러 소켓에서 흘러나오는 데이터 조각들을 모아, 각각에 송수신 주소 정보가 담긴 **헤더를 붙여서 하나의 하위 네트워크 계층으로 밀어 넣는 과정**입니다.

> **💡 핵심 식별자: 포트 번호 (Port Number)**
다중화와 역다중화를 가능하게 만드는 핵심 필드가 바로 세그먼트 헤더에 포함된 **출발지 포트 번호(Source Port)**와 **목적지 포트 번호(Destination Port)**입니다. 포트 번호는 16비트 크기(0 ~ 65535)를 가집니다.
> 

### 2. 비연결형 역다중화 (UDP의 방식)

UDP 소켓은 아주 단순하게 **'2-튜플(2-tuple)'** 정보로만 식별됩니다.

- **식별 기준:** `(목적지 IP 주소, 목적지 포트 번호)`

### ⚠️ UDP Demux의 특징과 결과

만약 출발지(Client)가 어디든 간에, **목적지 IP와 목적지 포트 번호만 같으면 커널은 전부 동일한 UDP 소켓으로 데이터를 던져줍니다.**

- **예시:** A 컴퓨터(포트 5000)와 B 컴퓨터(포트 6000)가 동시에 C 서버의 UDP 8000번 포트로 패킷을 보냈다면, C 서버 입장에서는 두 패킷 모두 **동일한 하나의 UDP 소켓**으로 받아보게 됩니다.
- 따라서 서버 애플리케이션은 패킷을 까보고 나서야 "아, 이것은 A가 보낸 거고, 저것은 B가 보낸 거구나"를 헤더의 출발지 IP/Port 정보를 보고 수동으로 분해해야 합니다.

### 3. 연결지향형 역다중화 (TCP의 방식)

TCP 소켓은 보안과 독립적인 통신 세션을 위해 훨씬 엄격한 **'4-튜플(4-tuple)'** 정보를 기준으로 식별됩니다.

- **식별 기준:** `(출발지 IP, 출발지 포트, 목적지 IP, 목적지 포트)`

### 🛠️ TCP Demux의 특징과 결과

목적지 IP와 포트가 완전히 같더라도, **출발지 IP나 출발지 포트 중 하나라도 다르면 커널은 완전히 다른 '독립된 TCP 소켓'으로 데이터를 안내**합니다.

- **예시 (웹 서버):** 전 세계 수만 명의 클라이언트가 포털 사이트의 `포트 80(HTTP)`으로 동시에 접속합니다.
- 웹 서버(Nginx, Apache 등)는 하나의 80번 포트를 열어두고 있지만, 커널 수준에서 클라이언트마다 고유한 4-튜플을 기반으로 **각기 다른 연결 소켓(Connection Socket)을 매핑**해 줍니다. 덕분에 사용자들의 데이터가 섞이지 않고 독립적인 스레드/프로세스에서 안전하게 처리됩니다.

| **구분** | **UDP 소켓 식별** | **TCP 소켓 식별** | **특징** |
| --- | --- | --- | --- |
| **식별자** | **2-Tuple** (Dst IP, Dst Port) | **4-Tuple** (Src IP, Src Port, Dst IP, Dst Port) | TCP가 더 촘촘하게 검사함 |
| **소켓 생성** | 목적지 포트당 1개 대기 | 연결(Handshake) 성공 시마다 신규 생성 | TCP는 다중 클라이언트 처리에 최적화 |

## 3.3 비연결형 트랜스포트:UDP

### 1. 왜 굳이 신뢰성이 없는 UDP를 사용할까?

데이터가 유실될 수도 있고, 순서가 뒤바뀔 수도 있는데 왜 TCP 대신 UDP를 사용할까요? 그 이유는 잃는 것(신뢰성)보다 얻는 것(속도와 제어권)이 훨씬 크기 때문입니다.

- **무자비한 전송 (애플리케이션의 정밀한 제어):** TCP는 네트워크가 혼잡해지면 패킷 유실을 막기 위해 스스로 전송 속도를 늦춥니다(혼잡 제어). 하지만 실시간 텔레메트리(원격 측정) 데이터나 조종 명령을 주고받을 때, 과거의 데이터가 조금 유실되더라도 가장 최신의 데이터를 지연 없이 빠르게 받아보는 것이 훨씬 중요합니다. UDP는 네트워크 상태를 신경 쓰지 않고 애플리케이션이 요구하는 즉시 데이터를 네트워크로 밀어 넣습니다.
- **연결 설정 지연이 없음 (No Connection Delay):** TCP는 데이터를 보내기 전에 3-way handshake라는 인사 과정을 거쳐야 합니다. 반면 비연결형인 UDP는 예비 동작 없이 즉각적으로 패킷을 발사합니다. (빠른 응답이 생명인 DNS가 UDP를 사용하는 가장 큰 이유입니다.)
- **가벼운 서버 상태 (No Connection State):** 연결 상태를 유지하지 않으므로, 서버는 버퍼 메모리나 혼잡 제어 파라미터를 추적할 필요가 없습니다. 덕분에 동일한 서버 리소스로 훨씬 더 많은 클라이언트의 트래픽을 동시에 감당할 수 있습니다.
- **초경량 헤더 오버헤드:** TCP는 패킷마다 20바이트의 무거운 헤더를 붙이지만, UDP 헤더는 단 **8바이트**에 불과합니다.

### 2. UDP 세그먼트 구조

UDP는 헤더가 단 4개의 필드(각 2바이트)로 이루어진 아주 단순한 구조를 가집니다.

| **출발지 포트 번호 (16비트)** | **목적지 포트 번호 (16비트)** |
| --- | --- |
| **길이 (16비트)** | **체크섬 (16비트)** |
| **애플리케이션 데이터 (페이로드)** |  |
- **길이(Length):** 헤더와 데이터를 합친 UDP 세그먼트 전체의 길이(바이트 단위)를 나타냅니다.
- 포트 번호는 데이터를 올바른 프로세스에게 전달(역다중화)하기 위한 최소한의 주소입니다.

### 3. 유일한 안전장치: 체크섬(Checksum)

UDP는 패킷이 목적지에 무사히 도착할지는 보장하지 않지만, "도착한 패킷 내부의 데이터가 깨지지는 않았는가?"는 최소한으로 검사합니다. 이를 체크섬(Checksum)이라고 합니다.

1. **송신 측:** 세그먼트 안의 모든 16비트 워드들을 더한 뒤, 1의 보수(비트 반전)를 취해 체크섬 필드에 집어넣습니다.
2. **수신 측:** 도착한 세그먼트의 모든 워드(체크섬 포함)를 다시 더합니다. 데이터 전송 중 비트 플립(0이 1로 바뀌는 등의 오류)이 없었다면 계산 결과는 모두 1(`1111111111111111`)이 나와야 합니다.
3. **오류 처리:** 만약 결과에 0이 하나라도 섞여 있다면 데이터가 전송 중 물리적 노이즈 등으로 훼손된 것입니다. UDP는 오류를 복구하려 하지 않고, **해당 패킷을 조용히 버려버립니다(Drop).**

> **💡 종단 간의 원칙 (End-to-End Principle)**
이더넷이나 Wi-Fi 같은 하위 링크 계층에도 오류 검사 기능이 있는데, 왜 트랜스포트 계층에서 체크섬을 또 할까요? 하위 계층의 검사는 '라우터와 라우터 사이'의 선로에서 발생하는 오류만 잡아낼 뿐, **라우터 내부의 메모리나 버퍼에서 발생하는 데이터 변조**는 잡아내지 못하기 때문입니다. 따라서 양 끝단(End)의 호스트가 최종적으로 한 번 더 검사해야 합니다.
>

## 3.4 신뢰적인 데이터 전송의 원리

하위 계층(네트워크 계층, IP)은 패킷을 분실하거나 비트를 뒤틀어버리는 '불신뢰적인 채널(Unreliable Channel)'입니다. 이 장의 핵심은 이 상처투성이인 바닥 위에서, 상위 애플리케이션 계층에게 어떻게 단 하나의 데이터 유실이나 오류도 없는 완벽한 '신뢰적인 채널(Reliable Channel)'을 제공할 것인가에 대한 프로토콜 설계 공학을 다룹니다.

### 1. rdt의 점진적 진화 과정

### rdt 1.0: 완벽하게 신뢰적인 채널

- **가정:** 하위 채널이 패킷을 절대 잃어버리지도 않고, 데이터가 깨지지도 않는 유토피아 같은 상황입니다.
- **구현:** 송신자는 주기만 하면 되고, 수신자는 받기만 하면 됩니다. 아무런 제어 로직이 필요 없습니다.

### rdt 2.0: 비트 오류(Corrupted)가 있는 채널

- **가정:** 패킷이 유실되지는 않지만, 전송 중에 비트가 0에서 1로 깨지는 오류가 발생할 수 있습니다. (마치 전화 통화 중 지지직거리는 잡음이 섞이는 상황)
- **해결책 (자동 재전송 요구, ARQ 메커니즘 도입):**
    1. **오류 검출:** UDP에서 배운 체크섬(Checksum)을 붙여 보냅니다.
    2. **수신자 피드백:** 잘 받았으면 **ACK(긍정 응답)**, 깨졌으면 NAK(부정 응답)를 송신자에게 답장합니다.
    3. **재전송:** 송신자는 NAK를 받으면 방금 보낸 패킷을 다시 보냅니다.
- **방식:** 패킷 하나를 보내고 응답이 올 때까지 다음 패킷을 보내지 않고 기다리는 **정지-대기(Stop-and-Wait)** 프로토콜입니다.

### ⚠️ rdt 2.0의 치명적인 결함과 rdt 2.1 / 2.2의 등장

- **문제점:** 만약 수신자가 보낸 **ACK나 NAK 패킷 자체가 전송 중에 깨지면** 송신자는 어떻게 해야 할까요? 잘 받았다는 건지 다시 달라는 건지 알 길이 없습니다. 멍하니 있을 수 없으니 송신자는 안전하게 패킷을 '재전송'하게 되는데, 이 경우 수신자는 이것이 **새로운 패킷인지, 아까 받았던 패킷의 중복(Duplicate)인지 구분할 수 없습니다.**
- **rdt 2.1 (순서 번호 도입):** 패킷 헤더에 순서 번호(Sequence Number)를 붙입니다. 정지-대기 방식이므로 번호는 0번과 1번, 단 2개만 있으면 됩니다. 수신자는 0번을 받았는데 또 0번이 오면 중복 패킷임을 감지하고 버린 뒤 ACK만 다시 던져줍니다.
- **rdt 2.2 (NAK 프리):** NAK를 없애고 ACK만 사용합니다. 1번 패킷이 깨졌다면 수신자는 "나 방금 전까지 0번은 잘 받았어"라는 의미로 `ACK 0`을 보냅니다. 송신자는 1번을 보냈는데 `ACK 0`이 오면 1번이 깨졌음을 인지하고 재전송합니다.

### rdt 3.0: 비트 오류 + 패킷 분실(Loss)이 있는 채널

- **가정:** 이제 데이터 비트가 깨질 뿐만 아니라, 패킷이 중간에 증발(유실)하거나 응답(ACK)이 영원히 사라질 수 있는 실제 인터넷 환경입니다.
- **문제점:** 송신자가 패킷을 보냈는데 중간에 증발하면, 수신자는 아무것도 못 받았으니 ACK를 안 보냅니다. 송신자는 정지-대기 상태이므로 ACK만 기다리며 두 호스트 모두 영원히 먹통(Deadlock)이 됩니다.
- **해결책 (카운트다운 타이머 도입):**
    - 송신자는 패킷을 보낼 때마다 타이머(Timer)를 가동합니다.
    - 설정한 시간(타임아웃, Timeout) 내에 ACK가 오지 않으면, 패킷이 공중에서 분실되었다고 판단하고 **패킷을 강제로 다시 전송**합니다.
    - 만약 패킷이 분실된 게 아니라 단순히 느리게 가던 중이어서 나중에 중복 ACK가 오더라도, 이미 rdt 2.1에서 구축한 '순서 번호' 덕분에 수신 측에서 중복 패킷을 안전하게 걸러낼 수 있습니다.

### 2. 성능의 한계와 파이프라이닝(Pipelining)

rdt 3.0은 완벽하게 작동하지만, 극도로 무능합니다. 패킷을 하나 던지고 지구 반대편에서 ACK가 올 때까지(RTT 동안) 하릴없이 기다려야 하기 때문입니다. 기가비트 광랜을 깔아두어도 대역폭의 1%도 쓰지 못하는 병목이 생깁니다.

이 효율성 문제를 해결하기 위해 **'정지-대기'를 버리고 '파이프라이닝(Pipelining)'** 방식으로 전환합니다. 응답을 기다리지 않고 수많은 패킷을 연속으로 밀어 넣는 방식입니다. 이때 연속으로 보낼 수 있는 패킷의 한계 범위를 '윈도우(Window)'라고 하며, 이를 구현하는 대표적인 두 가지 방식이 있습니다.

### A. Go-Back-N (GBN)

- 송신자는 ACK 없이 여러 패킷(예: 1~4번)을 연달아 보낼 수 있습니다.
- 수신자는 '누적 응답(Cumulative ACK)'을 씁니다. `ACK 3`을 보내면 "3번까지는 중간에 이빨 빠진 것 없이 완벽하게 다 잘 받았다"는 뜻입니다.
- **단점:** 만약 중간에 2번 패킷 하나가 유실되면, 수신자는 그 뒤에 도착하는 3, 4번 패킷을 모두 폐기합니다. 송신자는 2번 타이머가 터지는 순간, **성공적으로 전송되었던 3, 4번까지 포함하여 2번부터 그 뒤의 모든 패킷을 다시 다 보냅니다(Go Back).**

### B. 선택적 재전송 (Selective Repeat, SR)

- GBN의 비효율성을 보완하기 위해, 수신자는 중간에 이빨이 빠지더라도 도착한 패킷(예: 3, 4번)을 버리지 않고 자신의 버퍼에 보관합니다.
- 응답도 개별적으로 `ACK 2`, `ACK 3`을 따로 줍니다.
- 송신자는 **타임아웃이 발생한 '오직 그 유실된 패킷 하나만' 콕 집어서 재전송**합니다. 개별 패킷마다 타이머를 관리해야 하므로 구현이 더 복잡합니다.

아래의 시뮬레이터를 통해 rdt 3.0 환경에서 패킷 유실이 발생했을 때, 프로토콜이 타이머를 통해 어떻게 스스로 장애를 복구하고 재전송을 감행하는지 그 동적인 흐름을 직접 제어하며 확인해 볼 수 있습니다.
