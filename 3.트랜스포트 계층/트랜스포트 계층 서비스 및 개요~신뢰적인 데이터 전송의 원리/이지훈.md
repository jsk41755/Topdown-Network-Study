# 질문거리

## HTTP/2 멀티플렉싱(다중화) vs 트랜스포트 계층의 다중화/역다중화
| 구분 | HTTP/2 멀티플렉싱 | 트랜스포트 계층의 멀티플렉싱/역다중화 |
|---|---|---|
| 계층 | 애플리케이션 계층에 가까운 HTTP/2 프로토콜 수준 | 트랜스포트 계층, 예: TCP/UDP |
| 목적 | 하나의 TCP 연결 안에서 여러 HTTP 요청/응답을 동시에 처리 | 여러 애플리케이션 프로세스의 데이터를 네트워크로 보내고, 받은 데이터를 올바른 프로세스로 전달 |
| 핵심 단위 | 스트림(Stream), 프레임(Frame) | 포트 번호, 소켓 |
| 멀티플렉싱 의미 | 여러 HTTP 요청/응답 스트림을 하나의 TCP 연결에 섞어 보냄 | 여러 애플리케이션의 데이터를 하나의 네트워크 계층으로 내려보냄 |
| 역다중화 의미 | 받은 프레임을 Stream ID 기준으로 원래 HTTP 요청/응답에 재조립 | 받은 세그먼트/데이터그램을 목적지 포트 번호 기준으로 올바른 애플리케이션에 전달 |
| 식별자 | Stream ID | 포트 번호, IP 주소, 프로토콜, 소켓 |
| 동작 위치 | HTTP/2 연결 내부 | OS의 TCP/UDP 처리 영역 |
| 연결 관계 | 보통 하나의 TCP 연결 위에서 여러 HTTP 스트림 처리 | TCP/UDP가 여러 프로세스 통신을 구분 |
| 해결하려는 문제 | HTTP/1.1의 순차 처리, 연결 낭비, 요청 지연 감소 | 여러 네트워크 애플리케이션이 동시에 통신할 수 있게 함 |
| 예시 | 하나의 TCP 연결로 HTML, CSS, JS, 이미지 요청을 동시에 전송 | 브라우저는 443 포트 서버와 통신하고, 메신저 앱은 다른 포트/소켓으로 통신 |
| 한계 | TCP 위에서 동작하므로 TCP 레벨 Head-of-Line Blocking은 남음 | TCP 자체는 바이트 스트림이라 HTTP 요청 단위까지는 모름 |

핵심 차이는 HTTP/2 멀티플렉싱은 하나의 TCP 연결 안에서 여러 HTTP 스트림을 구분하는 것이고, 트랜스포트 계층의 멀티플렉싱/역다중화는 포트와 소켓을 이용해 여러 애플리케이션 통신을 구분하는 것

## 트랜트포트 계층의 다중화 vs 역다중화

| 구분 | 다중화 | 역다중화 |
|---|---|---|
| 일어나는 위치 | 송신 측 트랜스포트 계층 | 수신 측 트랜스포트 계층 |
| 방향 | 애플리케이션 계층 → 네트워크 계층 | 네트워크 계층 → 애플리케이션 계층 |
| 의미 | 여러 애플리케이션/소켓의 데이터를 모아 하나씩 세그먼트나 데이터그램으로 만들어 내려보냄 | 받은 세그먼트나 데이터그램을 알맞은 애플리케이션/소켓으로 나눠 전달 |
| 기준 정보 | 출발지 포트, 목적지 포트 등을 헤더에 붙임 | 헤더의 포트 번호와 IP 주소 등을 확인 |
| 핵심 역할 | 데이터에 식별 정보를 붙여 전송 가능하게 함 | 식별 정보를 보고 올바른 수신 소켓을 찾음 |
| 예시 | 브라우저, 게임, 메신저 데이터가 TCP/UDP를 통해 IP 계층으로 내려감 | 도착한 데이터가 포트 번호에 따라 브라우저, 게임, 메신저로 전달됨 |

> 다중화 = 여러 앱의 데이터를 하나로 모아 내려보내는 것
> 역다중화 = 받은 데이터를 다시 맞는 앱으로 나눠주는 것

예를 들어 네 컴퓨터에서 브라우저랑 게임이 동시에 인터넷을 쓴다고 하면, 보낼 때 TCP/UDP가 여러 앱의 데이터를 모아서 보내는 게 다중화고, 받을 때 도착한 데이터를 포트 번호를 보고 브라우저로 줄지 게임으로 줄지 나누는 게 역다중화.

## rdt 한 줄 요약
**rdt(reliable data transfer)** 는 “그냥 보내기” → “에러 검사” → “손실 대비 재전송” → “순서 보장/효율 개선”으로 발전했다고 보면 됨
### 1) rdt 1.0 — 세상은 완벽하다고 가정
- 비트 오류도 없고, 패킷 손실도 없음
- 그냥 보내면 끝
### 느낌
> “네트워크가 완벽하면 신경 쓸 게 없다”

현실적이지 않음.

### 2) rdt 2.0 — 비트 오류는 생길 수 있음
- 패킷 내용이 깨질 수 있다고 가정
- 그래서 checksum으로 오류 검사
- 오류가 있으면 NAK 보내고 재전송 요청
- 오류가 없으면 ACK 보냄

#### 문제
- ACK/NAK 자체도 깨질 수 있음
- 즉, “응답 메시지”가 신뢰할 수 없다는 문제가 생김.

### 3) rdt 2.1 — ACK/NAK도 깨질 수 있으니 번호를 붙이자
- 패킷에 **sequence number**를 붙임
- 같은 데이터를 중복으로 받았는지 구분 가능

#### 왜 필요?
예를 들어 ACK가 깨져서 송신자가 재전송하면, 수신자는 같은 패킷을 두 번 받는 상황이 생길 수 있음. 그래서
- 0번 패킷
- 1번 패킷

처럼 번갈아 구분해서 중복 처리함.

### 4) rdt 2.2 — NAK 없이 ACK만으로도 가능
- NAK를 따로 보내지 않고
- 그냥 “마지막으로 정상 수신한 패킷 번호”를 ACK로 보냄
- 송신자는 ACK가 예상과 다르면 재전송

#### 왜 바꾸나?
NAK 없이도 동작 가능해서 더 단순해짐.

### 5) rdt 3.0 — 손실까지 고려
- 이제는 패킷이 아예 사라질 수 있음을 가정.

그래서 추가로 필요한 것:
- **timer**
- 일정 시간 안에 ACK 못 받으면 재전송

### 핵심
오류뿐 아니라 유실(loss) 도 다뤄야 진짜 신뢰성 전송이 됨.

정리하면:

| 버전 | 고려하는 문제 | 해결 방법 |
|---|---|---|
| rdt 1.0 | 없음 | 그냥 전송 |
| rdt 2.0 | 비트 오류 | checksum, ACK/NAK |
| rdt 2.1 | ACK/NAK 오류, 중복 | sequence number |
| rdt 2.2 | NAK 없이 처리 | ACK만 사용 |
| rdt 3.0 | 패킷 손실까지 | timer + 재전송 |

#### 직관적으로 이해를 위한 rdt 진화 과정:

1. 에러 없다고 믿음
2. 에러 검사 추가
3. 중복/혼동 방지용 번호 추가
4. 손실 대비 타이머 추가
5. 더 나아가면 성능 개선을 위해 파이프라이닝으로 감

### 그 다음 단계: 성능 개선
rdt 3.0까지는 보통 한 번 보내고, 확인 오면 **다음 보내는 방식(stop-and-wait)**.

근데 이건 비효율적이어서, 이후에:
- GBN(Go-Back-N)
- SR(Selective Repeat)

같은 pipeline 방식이 나옴

### 차이 느낌
- Stop-and-wait: 하나 보내고 기다림
- GBN: 여러 개 보내고, 문제 생기면 그 이후 전부 다시
- SR: 문제난 것만 다시

#### 아주 쉽게 외우는 버전
> rdt는 “오류 없음”에서 시작해서, “오류 검사”, “중복 방지”, “손실 대응”을 차례로 추가한 신뢰성 전송 프로토콜의 진화 과정이다.
