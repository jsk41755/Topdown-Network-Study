## 4.1 네트워크 개요

### 1. 네트워크 계층의 두 가지 핵심 기능: 포워딩과 라우팅

네트워크 계층의 역할은 아주 명확하게 두 가지로 나뉜다. 이 둘의 차이를 명확히 구분하는 것이 4장과 5장 전체를 관통하는 핵심이다.

### A. 포워딩 (Forwarding / 전달)

- **정의:** 패킷이 라우터의 '입력 링크'로 들어왔을 때, 라우터 내부의 포워딩 테이블(Forwarding Table)을 참조하여 적절한 '출력 링크'로 패킷을 이동시키는 **단일 라우터 내부의 지역적(Local)인 행동**이다.
- **특징:** 나노초(Nanoseconds) 단위로 매우 짧은 시간에 일어나야 하므로, 라우터 내부의 하드웨어(실리콘 칩)에서 처리된다.
- **비유:** 자동차를 운전하다가 교차로에 진입했을 때, 이정표를 보고 "아, 부산 방향은 우회전이구나" 하고 핸들을 꺾어 빠져나가는 찰나의 행동과 같다.

### B. 라우팅 (Routing / 경로 설정)

- **정의:** 송신 호스트부터 수신 호스트까지 패킷이 거쳐가야 할 전 세계 네트워크의 **전체적인 경로(Route)를 계산하고 결정하는 네트워크 전체적(Global)인 행동**이다.
- **특징:** 라우팅 알고리즘을 통해 수많은 변수를 계산해야 하므로 상대적으로 긴 밀리초(Milliseconds)에서 초 단위로 실행되며, 주로 라우터의 소프트웨어(프로세서)에서 처리된다.
- **비유:** 출발 전 스마트폰 지도 앱을 켜서 서울에서 부산까지 갈 때 고속도로를 탈지, 국도를 탈지 **전체적인 여행 계획표**를 짜는 것과 같다.

> **💡 핵심 연계:** **라우팅(소프트웨어)** 알고리즘이 열심히 전 세계 지도를 계산해서 최적의 방향을 알아내면, 그 결과물을 각 라우터 하드웨어에 있는 **포워딩 테이블**에 적어준다. 그러면 라우터는 패킷이 들어올 때마다 묻지도 따지지도 않고 그 테이블만 보고 **포워딩(하드웨어)**을 수행한다.
> 

### 2. 데이터 평면(Data Plane)과 제어 평면(Control Plane)

위에서 설명한 두 기능은 현대 네트워크 아키텍처에서 두 개의 '평면(Plane)'으로 엄격하게 분리되어 설계된다.

- **데이터 평면 (Data Plane):** **포워딩**을 담당하는 영역이다. 오직 "들어온 패킷을 얼마나 빨리 내보낼 것인가?"에만 집중하는 근육(하드웨어) 역할이다. (교재 4장의 핵심 주제)
- **제어 평면 (Control Plane):** **라우팅**을 담당하는 영역이다. "포워딩 테이블을 어떻게 똑똑하게 채워 넣을 것인가?"를 고민하는 두뇌(소프트웨어) 역할이다. (교재 5장의 핵심 주제)

### 3. 네트워크 서비스 모델: 인터넷의 '최선형(Best-Effort)' 서비스

네트워크 계층을 설계할 때, 이론적으로는 상위 계층에게 다음과 같은 보장을 해줄 수 있다.

- **보장된 배달:** 패킷이 절대 유실되지 않음.
- **지연 시간 보장:** 패킷이 무조건 40ms 안에 도착함.
- **최소 대역폭 보장:** 1초에 최소 5Mbps의 속도를 무조건 뚫어줌.

하지만, **실제 인터넷의 네트워크 계층(IP 프로토콜)은 이 중 단 하나도 보장하지 않는다.** 이를 '최선형 서비스(Best-Effort Service)'라고 부른다.

- **최선형 서비스란?** "네가 보낸 패킷을 목적지까지 배달하기 위해 최선(Best-effort)은 다해볼게. 하지만 중간에 라우터 큐가 꽉 차서 패킷을 잃어버리거나(Loss), 순서가 뒤죽박죽되거나(Out-of-order), 언제 도착할지 시간(Delay)은 절대 장담 못 해!"라는 배짱 영업이다.
- **왜 이렇게 무책임하게 만들었을까?** 네트워크 코어(라우터들)를 극단적으로 **단순하고 멍청하게(Simple)** 유지하기 위해서이다.
    - 라우터가 대역폭을 보장하고 패킷 추적까지 하려면 인터넷 확장 비용이 천문학적으로 비싸진다.
    - 대신 이 복잡한 신뢰성 보장(재전송, 순서 맞추기, 흐름 제어)은 네트워크 끝단에 있는 똑똑한 PC들, 즉 트랜스포트 계층(TCP)에게 전부 떠넘긴 것이다. 이것이 인터넷이 전 세계로 폭발적으로 뻗어나갈 수 있었던 가장 위대한 설계 철학이다.

## 4.2 라우터 내부에는 무엇이 있을까?

### 1. 라우터의 4가지 핵심 구성 요소

라우터의 내부는 크게 4개의 블록으로 나눌 수 있다.

- **1) 입력 포트 (Input Ports):**
    - 패킷이 라우터로 들어오는 첫 관문이다. 물리적인 선이 꽂히는 곳이며, 1계층(물리)과 2계층(데이터 링크) 처리를 수행한다.
    - **가장 중요한 역할:** 라우팅 프로세서가 만들어준 '포워딩 테이블의 복사본'을 각 입력 포트가 스스로 가지고 있다. 패킷이 들어오면 중앙 CPU를 거치지 않고, 입력 포트 자체 하드웨어에서 초고속으로 테이블을 검색(Lookup)하여 어느 출력 포트로 갈지 결정한다.
- **2) 스위칭 구조 (Switching Fabric):**
    - 라우터의 심장이다. 입력 포트와 출력 포트를 연결하는 '라우터 내부의 네트워크(통로)'이다.
- **3) 출력 포트 (Output Ports):**
    - 스위칭 구조를 통해 넘어온 패킷들을 받아 물리적 링크를 통해 밖으로 쏘아 보내는 역할을 한다.
    - 패킷이 링크로 나가는 속도보다 스위칭 구조에서 넘어오는 속도가 더 빠르면, 출력 포트의 메모리에 패킷을 임시로 저장하는 버퍼링(큐잉)이 발생한다.
- **4) 라우팅 프로세서 (Routing Processor):**
    - 라우터의 두뇌(CPU)이다. 라우팅 프로토콜(OSPF, BGP 등) 소프트웨어를 돌려 전 세계 지도를 바탕으로 최적의 경로를 계산하고, 그 결과물인 포워딩 테이블을 만들어 각 입력 포트로 내려보낸다. (제어 평면의 역할)

### 2. 검색의 핵심 기술: 최장 프리픽스 매칭 (LPM)

입력 포트가 포워딩 테이블을 보고 패킷의 목적지를 찾을 때 사용하는 핵심 규칙이다.
라우터는 수십억 개의 개별 IP 주소를 다 외우지 않고, 주소의 대역폭(서브넷 범위) 단위로 묶어서 테이블에 저장한다.

- **최장 프리픽스 매칭 (Longest Prefix Matching):** 목적지 IP 주소와 포워딩 테이블의 항목들을 비교했을 때, 일치하는 앞부분(Prefix)의 길이가 **가장 긴(가장 구체적인)** 경로를 최종 목적지로 선택하여 패킷을 보낸다.

### 3. 스위칭 구조의 3가지 세대 (진화 과정)

라우터의 성능은 심장인 '스위칭 구조'가 얼마나 패킷을 빨리 넘겨주느냐에 달렸다. 역사적으로 3가지 방식이 사용되었다.

- **1세대 - 메모리 기반 (Memory):**
    - 초기 라우터(일반 PC와 유사)의 방식이다. 입력 포트로 들어온 패킷을 중앙 CPU의 메모리로 복사한 뒤, 다시 출력 포트로 복사한다.
    - 시스템 버스와 메모리 대역폭의 한계 때문에 한 번에 하나의 패킷만 이동할 수 있어 속도가 매우 느리다.
- **2세대 - 버스 기반 (Bus):**
    - 입력 포트와 출력 포트가 하나의 공유된 선(버스)으로 직접 연결된다. CPU를 거치지 않아 메모리 방식보다는 빠르다.
    - 하지만 통로가 하나뿐이라서, 한 번에 단 하나의 패킷만 버스를 탈 수 있다. 버스의 대역폭이 전체 라우터의 처리 용량을 제한하는 병목 현상이 발생한다.
- **3세대 - 상호연결 네트워크 / 크로스바 (Crossbar):**
    - 현대의 고성능 라우터가 사용하는 방식이다. 가로세로 격자망(Grid)을 만들어 교차점에 반도체 스위치를 달아놓은 구조이다.
    - A포트에서 B포트로, C포트에서 D포트로 가는 패킷의 동선이 겹치지 않는다면, **여러 개의 패킷을 동시에 병렬로 전송**할 수 있어 압도적인 속도를 자랑한다.

### 4. 라우터 안의 병목 현상: HOL 블로킹

라우터 내부에서도 패킷이 막히거나 유실될 수 있다. 가장 대표적인 현상이 하드웨어 버퍼에서 발생하는 **입력 포트 큐잉과 HOL 블로킹**이다.

- **HOL 블로킹 (Head-of-Line Blocking):** 입력 포트의 대기열(큐) 맨 앞에 있는 패킷이 가려는 출력 포트가 현재 다른 패킷에 의해 사용 중이라서 막혀 있는 상황을 가정해 보자. 이때 대기열 두 번째에 있는 패킷은 자기가 가려는 출력 포트가 비어 있음에도 불구하고, 맨 앞의 패킷이 길을 막고 있어서 나가지 못하고 꼼짝없이 갇히게 되는 현상이다.
- 이는 3.8절에서 배운 TCP의 HOLB와 이름은 같지만, 발생하는 위치(라우터의 물리적 메모리 구조 vs 애플리케이션 바이트 스트림)가 전혀 다르다는 점을 구별해야 한다.

### 5. 큐잉(Queuing)은 어디서 일어나는가?

패킷이 라우터 안에서 줄을 서서 기다리는 현상은 입력 포트와 출력 포트 모두에서 발생할 수 있다.

- **입력 큐잉 (Input Queuing):** * 스위칭 구조의 속도가 너무 느려 여러 입력 포트에서 들어오는 패킷을 제때 처리하지 못할 때 발생한다. 앞서 언급한 **HOL 블로킹**이 대표적인 입력 큐잉의 부작용이다.
- **출력 큐잉 (Output Queuing):**
    - 스위칭 구조가 패킷을 넘겨주는 속도가 출력 링크의 전송 속도보다 빠를 때 발생한다. 현대의 고성능 라우터에서는 스위칭 속도가 매우 빠르기 때문에, 대부분의 패킷 지연과 유실은 이 출력 포트의 큐(버퍼)에서 일어난다.

### 6. 패킷 폐기 정책 (Packet Drop Policies)

출력 큐(메모리 Buffer)가 가득 찼을 때, 새로 들어오는 패킷을 어떻게 처리할지 결정하는 규칙이다.

- **꼬리 폐기 (Tail Drop):**
    - 가장 단순한 방식이다. 큐가 꽉 차면 그 이후에 도착하는 모든 패킷을 무조건 버린다.
- **능동 큐 관리 (AQM, Active Queue Management):**
    - 큐가 완전히 차기 전에 버퍼의 평균 길이를 모니터링하여, 선제적으로 패킷을 버리는 방식이다. 가장 대표적인 알고리즘이 RED(Random Early Detection)이다.
    - 큐가 일정 임계값을 넘어가면 확률적으로 패킷을 골라 미리 버림으로써, TCP 송신자들이 눈치를 채고 스스로 전송 속도를 줄이도록 유도하여 대규모 혼잡을 예방한다.

### 7. 패킷 스케줄링 정책 (Packet Scheduling Disciplines)

버퍼에 쌓여 있는 여러 패킷 중 "다음에 출력 링크로 내보낼 패킷은 무엇인가?"를 결정하는 규칙이다. 이는 네트워크의 공평성과 서비스 품질(QoS)을 결정한다.

- **1) FIFO (First-In-First-Out):**
    - 먼저 도착한 패킷을 먼저 내보내는 가장 기본적인 방식이다. 패킷의 중요도나 종류를 전혀 고려하지 않는다. (FCFS 방식과 동일)
- **2) 우선순위 스케줄링 (Priority Scheduling):**
    - 패킷들을 중요도에 따라 클래스(예: 고우선순위, 저우선순위)로 분류한다. **고우선순위 큐에 패킷이 단 한 개라도 남아 있다면, 저우선순위 큐의 패킷은 절대 내보내지 않는다.**
    - 실시간 제어 데이터나 보이스(VoIP) 트래픽을 우대할 때 좋지만, 고우선순위 트래픽이 몰리면 저우선순위 트래픽은 영원히 굶주리는 기아 현상(Starvation)이 발생한다.
- **3) 라운드 로빈 (Round Robin):**
    - 패킷들을 클래스별로 분리된 큐에 넣은 뒤, 스케줄러가 순서대로 회전하며 각 큐에서 패킷을 1개씩 번갈아 가며 꺼내 보낸다. 기아 현상을 완벽히 방지한다.
- **4) WFQ (Weighted Fair Queuing / 가중 공정 큐잉):**
    - 현대 라우터에서 가장 널리 쓰이는 고도화된 방식이다. 라운드 로빈을 발전시켜, **각 클래스(큐)마다 서로 다른 '가중치(Weight)'를 부여**한다.
    - 예를 들어 1번 큐의 가중치가 3이고 2번 큐가 1이라면, 전체 대역폭을 3:1의 비율로 공평하게 나누어 전송한다. 특정 트래픽이 대역폭을 독점하는 것을 막고 최저 속도를 보장할 수 있다.

## 4.3 인터넷 프로토콜(IP): IPv4, 주소체계, IPv6 등

### 1. IPv4 데이터그램 포맷

네트워크 계층의 패킷을 데이터그램(Datagram)이라고 부른다. 트랜스포트 계층(TCP/UDP)에서 내려온 세그먼트에 IP 헤더가 붙어 완성된다.

- **헤더 크기:** 기본적으로 **20바이트**이다. (만약 TCP를 사용한다면 TCP 헤더 20바이트를 더해, 순수 데이터 외에 최소 40바이트의 오버헤드가 발생한다.)
- **주요 핵심 필드:**
    - **수명 (TTL, Time-To-Live):** 네트워크에서 패킷이 영원히 떠도는(Loop) 것을 막기 위한 타이머이다. 라우터를 하나 거칠 때마다 1씩 감소하며, 0이 되면 라우터가 패킷을 즉시 폐기한다.
    - **상위 계층 프로토콜:** 이 데이터그램의 페이로드가 TCP인지(6번) UDP인지(17번) 명시하여, 수신 호스트가 올바른 트랜스포트 계층으로 넘길 수 있게 한다.
    - **헤더 체크섬:** 라우터 내부에서 IP 헤더의 비트 오류가 발생했는지 검사한다.

### 2. IPv4 주소체계와 CIDR

IPv4 주소는 32비트로 구성되며, 총 개(약 43억 개)의 주소를 가질 수 있다. 라우터 없이 물리적으로 직접 통신할 수 있는 장치들의 집합을 서브넷(Subnet)이라고 부른다.

- **CIDR (Classless Inter-Domain Routing):**
    - 과거의 비효율적인 클래스(A, B, C) 방식을 폐기하고 도입한 유연한 주소 할당 방식이다.
    - IP 주소를 `192.168.1.0/24`와 같이 표기한다. 여기서 `/24`를 서브넷 마스크(Prefix)라고 부르며, 32비트 중 앞의 24비트는 '네트워크 주소'로, 나머지 8비트는 '호스트 주소'로 사용한다는 뜻이다.
    - 서브넷 마스크를 조절하여 필요한 만큼만 IP 주소를 잘게 쪼개어 할당할 수 있다.

### 3. IP 주소를 얻는 방법: DHCP

컴퓨터나 스마트폰이 인터넷에 접속하려면 IP 주소가 필요하다. 이를 자동으로 할당해 주는 것이 DHCP (Dynamic Host Configuration Protocol)이다. 플러그 앤 플레이(Plug-and-play)를 가능하게 해주는 고마운 프로토콜이다.

- **동작 과정 (DORA 과정):**
    1. **Discover (발견):** 클라이언트가 "여기 DHCP 서버 있나요?"라며 브로드캐스트 패킷을 뿌린다.
    2. **Offer (제공):** DHCP 서버가 "내가 IP 주소 하나 빌려줄게"라며 응답한다.
    3. **Request (요청):** 클라이언트가 "그 주소 제가 쓸게요!"라고 확정 요청을 보낸다.
    4. **ACK (확인):** 서버가 최종 승인하며 IP 주소, 서브넷 마스크, 기본 게이트웨이, DNS 서버 주소를 함께 넘겨준다.

### 4. IPv4의 생명 연장: NAT (네트워크 주소 변환)

43억 개의 IPv4 주소는 이미 2011년경에 모두 고갈되었다. 그럼에도 우리가 인터넷을 잘 쓰고 있는 이유는 바로 공유기에 탑재된 **NAT (Network Address Translation)** 덕분이다.

- **원리:**
    - 가정이나 회사의 공유기 내부에 있는 기기들(스마트폰, PC, TV)에게는 **사설 IP**(예: `192.168.0.x`)를 부여한다. 이 사설 IP는 외부 인터넷에서는 통용되지 않는 가짜 주소다.
    - 이 기기들이 외부로 패킷을 보낼 때, 공유기가 자신의 유일한 **공인 IP**로 주소를 몰래 덧씌워(Translation) 내보낸다.
- **핵심 트릭 (포트 번호의 활용):** 여러 대의 사설 IP 기기가 하나의 공인 IP를 어떻게 공유할까? 공유기의 NAT 변환 테이블은 IP 주소뿐만 아니라 '포트 번호(Port Number)'까지 함께 변환하고 기록한다. 돌아오는 패킷의 포트 번호를 보고 어떤 내부 기기에게 줄지 완벽하게 구분해 낸다.

### 5. 근본적 해결책: IPv6

NAT가 훌륭한 미봉책이긴 하지만, 네트워크 구조를 복잡하게 만들고 종단 간(End-to-End) 연결 원칙을 위배한다는 단점이 있다. 이를 근본적으로 해결하기 위해 등장한 것이 **IPv6**이다.

- **핵심 변경점:**
    - **128비트 주소: $2^{128}$**개의 주소 공간을 제공한다. (지구상의 모든 모래알에 IP를 부여하고도 남는 양이다.)
    - **고정 크기 40바이트 헤더:** IPv4의 복잡한 옵션 필드를 전부 빼버리고 헤더 크기를 고정하여, 라우터가 패킷을 처리(포워딩)하는 속도를 비약적으로 높였다.
    - **체크섬(Checksum) 제거:** 2계층(데이터 링크)과 4계층(트랜스포트)에서 이미 체크섬을 하고 있기 때문에, 처리 속도를 높이기 위해 3계층 헤더에서는 검사를 아예 과감하게 삭제했다.
- **전환 기술 (터널링, Tunneling):** 전 세계의 모든 IPv4 라우터를 하루아침에 IPv6로 바꿀 수는 없다. 따라서 두 대의 IPv6 라우터가 통신할 때, 그 사이에 있는 IPv4 망을 지나기 위해 **IPv6 데이터그램 전체를 IPv4 데이터그램의 '페이로드' 안에 캡슐처럼 쏙 집어넣어** 전송한다. 이를 터널링이라고 한다.

## 4.4 일반화된 포워딩 및 소프트웨어 기반 네트워크(SDN)

### 1. 매치 플러스 액션 (Match-plus-Action) 패러다임

전통적인 라우터의 포워딩 테이블은 오직 `[목적지 IP 주소 범위 -> 출력 링크]`라는 단순한 구조였다. 하지만 SDN이 도입한 '일반화된 포워딩(Generalized Forwarding)'은 이를 플로우 테이블(Flow Table)이라는 개념으로 확장했다.

플로우 테이블은 매치(Match)와 액션(Action)이라는 두 가지 강력한 무기를 가진다.

- **매치 (Match 조건을 아주 다양하게!):**
    - 패킷이 들어왔을 때, 목적지 IP 주소뿐만 아니라 **출발지 IP, MAC 주소, TCP/UDP 포트 번호, 심지어 VLAN ID까지** 수많은 헤더 필드를 조합하여 검사할 수 있다.
    - (예: "목적지 IP가 10.0.0.1이고, TCP 목적지 포트가 80(웹)인 패킷들만 골라내라!")
- **액션 (Action을 마음대로!):**
    - 매치된 패킷을 단순히 1번 링크로 보내는(Forward) 것을 넘어, 패킷을 조용히 **버려버리거나(Drop)**, 패킷의 IP나 포트 번호를 실시간으로 수정(Modify)해서 내보낼 수도 있다.

### 2. 일반화된 포워딩이 만들어내는 마법 (가상화)

이 매치 플러스 액션 구조 덕분에, 하나의 멍청한 스위치(SDN 스위치)가 플로우 테이블의 규칙(Rule)을 어떻게 설정하느냐에 따라 **네트워크의 어떤 장비로든 자유자재로 변신**할 수 있게 되었다.

- **라우터(Router)로 변신:** 매치를 `목적지 IP 주소`로, 액션을 `출력 포트로 포워딩`으로 설정하면 평범한 라우터가 된다.
- **스위치(Switch)로 변신:** 매치를 `목적지 MAC 주소`로, 액션을 `포워딩`으로 설정하면 2계층 이더넷 스위치가 된다.
- **방화벽(Firewall)으로 변신:** 매치를 `특정 IP 대역`이나 `포트 번호(예: 22번 SSH)`로, 액션을 `Drop(폐기)`으로 설정하면 훌륭한 방화벽이 된다.
- **NAT(네트워크 주소 변환기)로 변신:** 매치를 `특정 IP 대역`으로, 액션을 `IP 주소 및 포트 번호 수정(Modify)`으로 설정하면 공유기의 NAT 기능을 완벽히 수행한다.

### 3. 미들박스(Middlebox)의 등장과 네트워크 기능 가상화(NFV)

과거에는 방화벽, NAT, 로드 밸런서 등 각각의 특수한 기능을 수행하기 위해 기업들은 수천만 원짜리 전용 하드웨어 장비(미들박스)를 종류별로 따로 구매하여 네트워크 중간중간에 설치해야 했다. 이는 관리를 지옥으로 만들고 확장성을 크게 떨어뜨렸다.

- **SDN과 NFV(Network Functions Virtualization)의 결합:**
    - 이제는 복잡한 전용 장비를 살 필요가 없다. 텅 빈 깡통 같은 일반 서버(Commodity Server)와 저렴한 SDN 스위치를 수십 대 깔아둔다.
    - 그리고 중앙의 SDN 컨트롤러(소프트웨어 두뇌)에서 1번 스위치에는 방화벽 플로우 테이블을, 2번 스위치에는 로드 밸런서 플로우 테이블을 내려보내어(프로그래밍하여) 네트워크를 소프트웨어적으로 자유롭게 구성(Virtualization)한다.
    - 클라우드 데이터센터(AWS, Google Cloud 등)가 이 기술 덕분에 수만 명의 고객에게 클릭 몇 번만으로 완벽히 격리된 가상 네트워크 환경(VPC)을 순식간에 만들어 줄 수 있는 것이다.

## 4.5 미들박스

교재 4.5절 '미들박스(Middleboxes)'는 인터넷의 기본 설계 철학인 '단순한 네트워크 코어와 똑똑한 종단(End-to-End 원칙)'을 흔들며 등장한, 현대 네트워크의 필수 불결한 존재인 다양한 전용 장비들을 다루는 단원이다.

전통적인 라우터가 오직 원본 패킷의 IP 주소만 보고 길을 찾아주는 역할만 수행한다면, 미들박스는 "송신 호스트와 수신 호스트 사이의 데이터 경로 상에서 IP 라우팅 이외의 기능을 수행하는 모든 네트워크 장비"를 통칭한다.

### 1. 미들박스의 정의와 출현 배경

인터넷은 원래 하위 라우터들이 패킷 내부를 들여다보지 않고 오직 전달만 하는 단순한 구조로 설계되었다. 그러나 인터넷이 급격히 성장하면서 보안 공격, IP 주소 부족, 트래픽 폭주 등의 현실적인 문제가 발생했다. 이를 해결하기 위해 라우터와 라우터 사이에 특수한 목적을 가진 ' 중간 장비'들이 끼어들기 시작했는데, 이를 미들박스라고 부른다.

### 2. 대표적인 미들박스의 종류

미들박스는 동작하는 네트워크 계층과 목적에 따라 다양하게 분류된다.

- **방화벽 (Firewall):**
    - 패킷의 헤더(IP, 포트)나 페이로드를 검사하여 허가되지 않은 위험한 트래픽을 차단(Drop)하는 보안 장비이다.
- **NAT (네트워크 주소 변환기):**
    - 앞서 4.3절에서 배운 장비이다. 사설 IP 주소를 공인 IP 주소로 변환하여 IP 주소 고갈 문제를 해결한다. 패킷의 IP와 포트 번호를 실시간으로 수정하는 미들박스의 일종이다.
- **부하 분산기 (Load Balancer):**
    - 하나의 웹 서비스로 들어오는 수많은 클라이언트의 요청을 여러 대의 실제 백엔드 서버로 공평하게 나누어 전송(Divert)하여 서버의 과부하를 막는 장비이다.
- **DPI (Deep Packet Inspection / 심층 패킷 검사기):**
    - 패킷의 상위 애플리케이션 계층 페이로드 데이터까지 샅샅이 뒤져서 악성코드 유무를 확인하거나, 특정 프로토콜(예: 토렌트 등)의 사용을 차단하는 고도화된 검사 장비이다.

### 3. 전통적 하위 미들박스의 한계와 문제점

미들박스는 네트워크를 안전하고 효율적으로 만들어 주었지만, 다음과 같은 심각한 아키텍처적 한계를 유발했다.

- **종단 간 원칙(End-to-End Principle)의 훼손:** 양 끝단의 호스트가 데이터를 온전히 주고받아야 한다는 인터넷의 기본 원칙을 깨뜨린다. 중간에서 패킷을 변조(NAT)하거나 임의로 차단(방화벽)하기 때문이다.
- **독점적 하드웨어의 높은 비용:** 과거에는 방화벽, NAT 등을 수행하기 위해 시스코, 주니퍼 등 장비 제조사가 만든 비싼 전용 하드웨어를 각각 따로 구매해야 했다.
- **경직된 토폴로지 구조 (지옥 같은 관리):** 새로운 미들박스를 추가하려면 네트워크 선을 물리적으로 끊고 중간에 새 장비를 끼워 넣어야 했다. 트래픽이 `[방화벽 -> NAT -> 부하분산기]` 순서로 흐르도록 물리적 배선을 맞추는 작업은 매우 복잡하고 유연성이 떨어졌다.

### 4. 현대적 진화: 가상 미들박스와 서비스 체이닝

이러한 하드웨어 미들박스의 한계를 극복하기 위해 등장한 개념이 바로 4.4절에서 다룬 SDN(소프트웨어 정의 네트워크)과 NFV(네트워크 기능 가상화)이다.

현대 데이터센터에서는 미들박스를 비싼 물리 장비로 두지 않는다. 일반 고성능 서버 위에 방화벽 소프트웨어, NAT 소프트웨어를 가상 머신(VM)이나 컨테이너 형태로 구동한다. 그리고 SDN 컨트롤러를 통해 플로우 테이블 규칙을 내려보내어, 패킷이 하드웨어 선 연결 없이도 소프트웨어적으로 `[방화벽 VM -> 부하분산기 VM]` 순서로 알아서 거쳐 가도록 제어한다. 이를 '서비스 체이닝(Service Chaining)'이라고 부른다.

다양한 종류의 트래픽 패킷을 흘려보내며, 중간에 배치된 미들박스들(방화벽, 부하 분산기, NAT)의 활성화 상태와 규칙 변경에 따라 패킷이 어떻게 필터링되고, 분산되고, 주소가 변환되어 흘러가는지 다음 서비스 체이닝 시뮬레이터를 통해 직관적으로 확인할 수 있다.
