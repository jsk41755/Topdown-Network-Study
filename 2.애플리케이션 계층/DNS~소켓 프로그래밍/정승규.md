### 1. DNS(Domain Name System)의 핵심 역할

사람은 `www.google.com`이나 `naver.com`처럼 기억하기 쉬운 문자 형태의 호스트 이름(Hostname)을 선호한다. 하지만 인터넷 코어의 라우터들은 데이터를 전달할 때 고정된 길이의 숫자로 이루어진 IP 주소(IP Address)만을 이해한다.

이 둘 사이의 간극을 메우기 위해 호스트 이름을 IP 주소로 변환해 주는 인터넷의 '전화번호부' 같은 역할이 필요한데, 이것이 바로 **DNS**이다.

DNS는 단순히 주소 변환만 하는 것이 아니라 다음과 같은 추가 서비스도 제공한다.

- **호스트 에일리어싱 (Host Aliasing):** 복잡하고 긴 정식 이름(Canonical hostname) 대신 기억하기 쉬운 별명(Alias)을 사용할 수 있게 해준다.
- **메일 서버 에일리어싱 (Mail Server Aliasing):** 웹 서버와 메일 서버가 같은 별명(예: `yahoo.com`)을 사용할 때, 메일 서버의 진짜 주소를 찾아준다.
- **부하 분산 (Load Distribution):** 트래픽이 많은 사이트는 여러 대의 서버(여러 IP 주소)가 하나의 호스트 이름을 공유한다. DNS는 요청이 올 때마다 이 IP 주소들의 순서를 바꿔가며 응답하여 트래픽을 분산시킨다.

### 2. 분산 계층 데이터베이스 구조

**왜 전 세계의 모든 주소를 하나의 중앙 DNS 서버에 모아두지 않을까?**
만약 단일 서버가 전 세계의 모든 트래픽을 처리한다면 서버가 다운될 경우 인터넷 전체가 마비(단일 고장점, Single Point of Failure)되고, 트래픽 폭주와 원거리 접속으로 인한 극심한 지연이 발생하기 때문이다.

따라서 DNS는 전 세계에 분산된 서버들을 계층적(Hierarchical)으로 구성한다.

1. **루트 DNS 서버 (Root DNS Servers):** 가장 최상위에 위치하며, 사용자가 검색하려는 도메인의 확장자(.com, .kr 등)를 담당하는 TLD 서버의 IP 주소를 알려준다.
2. **TLD (Top-Level Domain) 서버:** `.com`, `.org`, `.net` 같은 일반 도메인과 `.kr`, `.uk` 같은 국가 도메인을 담당한다. 해당 도메인에 속한 기관의 책임 서버 주소를 알려준다.
3. **책임 DNS 서버 (Authoritative DNS Servers):** 대학, 기업 등 인터넷에 호스트를 가진 기관이 직접 관리하거나 호스팅 업체를 통해 관리하는 서버이다. 자신들의 호스트 이름에 대한 '최종적인(실제) IP 주소'를 가지고 있다.
4. **로컬 DNS 서버 (Local DNS Server):** 엄밀히 말해 위 3단계 계층 구조에 속하지는 않지만, 핵심적인 역할을 한다. ISP(KT, SKT 등)나 학교, 기업 네트워크마다 존재하며, 사용자의 호스트가 가장 먼저 쿼리(질의)를 보내는 대리인(프록시) 서버이다.

### 3. DNS 질의(Query) 작동 방식: 재귀적 vs 반복적

사용자가 브라우저에 `gaia.cs.umass.edu`를 입력했을 때, DNS가 IP 주소를 찾아오는 과정은 두 가지 방식이 결합되어 있다.

- **재귀적 질의 (Recursive Query):** "내가 못 찾으면 네가 대신 끝까지 찾아서 정답만 알려줘!"
- 주로 **사용자의 PC(호스트) ↔ 로컬 DNS 서버** 구간에서 일어난다. 호스트는 로컬 서버에 요청을 던지고 정답이 올 때까지 기다린다.
- **반복적 질의 (Iterative Query):** "내가 모르면, 누구한테 물어봐야 하는지 그 주소만 알려줄게. 네가 직접 다시 물어봐!"
- 주로 **로컬 DNS 서버 ↔ 루트/TLD/책임 서버** 구간에서 일어난다.
- 로컬 서버가 루트 서버에 물어보면, 루트는 "난 모르지만 `.edu` TLD 서버한테 가봐"라고 주소를 준다.
- 로컬 서버가 다시 `.edu` TLD 서버한테 물어보면, "난 모르지만 `umass.edu` 책임 서버한테 가봐"라고 주소를 준다.
- 최종적으로 책임 서버가 진짜 IP 주소를 로컬 서버에게 반환한다.

### 4. 웹 속도를 살려주는 핵심: DNS 캐싱 (Caching)

위처럼 매번 계층을 오르내리며 질문하는 것은 네트워크 지연을 유발한다. 이를 해결하기 위해 DNS는 **캐싱**을 적극적으로 활용한다.

로컬 DNS 서버는 다른 서버로부터 응답을 받을 때마다 그 정보(호스트 이름 ↔ IP 매핑)를 자신의 메모리에 임시로 저장(캐시)해 둔다.
만약 다른 사용자가 조금 뒤에 똑같은 호스트 이름을 요청하면, 로컬 서버는 루트나 TLD 서버에 묻지 않고 캐시된 IP 주소를 즉시 반환한다. 데이터가 영원히 남아있는 것을 방지하기 위해 각 데이터에는 수명(TTL, Time to Live)이 설정되어 있어 시간이 지나면 삭제된다.

### 5. DNS 레코드와 메시지

DNS 서버들이 주고받는 데이터는 자원 레코드(RR, Resource Records)라는 형식으로 저장되며, 기본 포맷은 `(Name, Value, Type, TTL)`이다. 가장 중요한 Type 4가지는 다음과 같다.

- **Type A:** Name은 호스트 이름, Value는 IP 주소이다. (가장 기본적인 주소 변환)
- 예: `(relay1.bar.foo.com, 145.37.93.126, A)`
- **Type NS:** Name은 도메인, Value는 해당 도메인을 담당하는 책임 DNS 서버의 호스트 이름이다.
- 예: `(foo.com, dns.foo.com, NS)`
- **Type CNAME:** Name은 별명, Value는 진짜 정식 호스트 이름이다.
- **Type MX:** Name은 메일 서버의 별명, Value는 진짜 정식 메일 서버 이름이다.

**DNS 프로토콜의 특징:**
DNS 쿼리와 응답 메시지는 하위 트랜스포트 계층으로 UDP (포트 번호 53)를 사용한다. TCP의 연결 설정(핸드셰이킹) 오버헤드를 없애 속도를 극대화하기 위함이다. UDP는 신뢰성이 없지만, DNS는 응답이 오지 않으면 애플리케이션(브라우저 등) 단에서 짧은 타임아웃 후 재요청을 보내는 방식으로 이를 보완한다.

## 2.5 P2P 파일 분배

**1. 클라이언트-서버(C-S) vs P2P 확장성 비교**
하나의 서버가 크기 F인 파일을 N개의 피어(사용자)에게 분배해야 하는 상황을 가정한다.

- $u_s$: 서버의 접속 링크 업로드 속도
- $d_i$: $i$번째 피어의 다운로드 속도( $d_{min}$은 이 중 가능 느린 속도)
- $u_i$: $i$번째 피어의 업로드 속도

**A. 클라이언트-서버(C-S) 구조의 분배 시간 (**$D_{cs}$**)**
서버는 파일 복사본을 N번 전송해야 하므로 총 NF 비트를 업로드해야 한다.

$D_{cs} \ge \max \left\{ \frac{NF}{u_s}, \frac{F}{d_{min}} \right\}$

- **치명적 단점:** 사용자의 수 N이 증가할수록 분배 시간이 **선형적으로 무한히 증가**한다. 파일 배포가 전적으로 서버의 업로드 대역폭($u_s$)에만 의존하기 때문이다.

**B. P2P 구조의 분배 시간 (**$D_{p2p}$**)**
P2P에서는 서버가 파일을 단 한 번만 네트워크에 업로드($F/u_s$)하면 된다. 이후에는 파일을 다운로드받은(혹은 받고 있는) 피어들이 자신의 업로드 속도($u_i$)를 활용해 다른 피어들에게 파일을 전송한다. 전체 시스템의 최대 업로드 용량은 서버와 모든 피어의 업로드 용량의 합인 $u_s + \sum u_i$가 된다.
$D_{p2p} \ge \max \left\{ \frac{F}{u_s}, \frac{F}{d_{min}}, \frac{NF}{u_s + \sum u_i} \right\}$

- **자가 확장성 (Self-Scalability):** $N$이 증가하여 요구되는 전체 데이터양(NF)이 늘어나더라도, 각 피어들이 새로운 업로드 용량($u_i$)을 시스템에 추가로 제공하므로 분배 시간이 크게 늘어나지 않고 완만하게 유지된다.

### 2. 실제 P2P 애플리케이션: 비트토렌트 (BitTorrent)

교재에서는 P2P 파일 분배의 가장 성공적인 실제 사례로 비트토렌트의 구조와 핵심 알고리즘 두 가지를 소개한다.

**기본 구조 용어:**

- **토렌트(Torrent):** 특정 파일의 청크(조각)들을 교환하는 피어들의 집합이다.
- **트래커(Tracker):** 토렌트에 참여하는 피어들의 IP 주소를 추적하고 관리하는 인프라 서버이다. (파일 데이터를 직접 가지고 있지는 않다.)
- **청크(Chunk):** 비트토렌트는 파일을 보통 **256KB** 크기의 균등한 조각(청크) 단위로 나누어 주고받는다.

### 핵심 알고리즘 1: 가장 드문 것 먼저 (Rarest First)

피어는 이웃들(자신과 연결된 다른 피어들)이 어떤 청크를 가지고 있는지 주기적으로 목록을 교환한다. 자신이 필요한 청크 중, **이웃들 사이에서 가장 적게 존재하는(가장 희귀한) 청크를 최우선으로 요청**한다.

- **효과:** 희귀한 청크가 네트워크에서 사라지는 것을 방지하고, 파일의 모든 조각이 네트워크 내에 고르게 퍼지도록 하여 병목 현상을 막는다.

### 핵심 알고리즘 2: 팃포탯 (Tit-for-Tat / 이에는 이, 눈에는 눈)

P2P 시스템은 사용자가 다운로드만 받고 업로드는 하지 않는 '무임승차(Free-riding)'를 막아야 한다. 비트토렌트는 자신에게 가장 잘해주는 사람에게만 보답하는 이기적인 스케줄링 알고리즘을 사용한다.

1. **Choking/Unchoking:** 자신에게 가장 높은 속도로 데이터를 보내주는 상위 4명의 피어를 평가하여, 이 4명에게만 자신의 데이터를 보낸다(Unchoked). 나머지 피어들에게는 데이터 전송을 차단한다(Choked). 이 평가는 10초마다 갱신된다.
2. **낙관적 차단 해제 (Optimistic Unchoking):** 만약 기존 4명하고만 통신한다면 더 나은 파트너를 영원히 찾을 수 없다. 따라서 **30초마다 무작위로 새로운 피어 한 명을 선택**하여 아무 조건 없이 데이터를 보낸다.
- 만약 이 무작위 피어가 더 빠른 속도로 보답해 준다면, 기존의 4명 중 가장 느린 피어를 밀어내고 상위 4명 그룹에 새로 편입된다.
- 이 메커니즘을 통해 네트워크 전체의 피어들이 서로 최적의 전송 파트너를 지속적으로 찾아갈 수 있다.

## 2.6 비디오 스트리밍과 콘텐츠 분배 네트워크

### 1. 인터넷 비디오의 특성과 압축

비디오 스트리밍의 가장 큰 난관은 데이터의 크기이다. 이를 해결하기 위해 비디오는 고도의 압축 기술을 사용한다.

- **공간적 중복성 (Spatial Redundancy):** 한 프레임 내에서 같은 색상의 배경(예: 파란 하늘)이 반복될 때, 모든 픽셀의 색상을 기록하지 않고 "여기부터 저기까지 파란색"이라는 식으로 압축한다.
- **시간적 중복성 (Temporal Redundancy):** 연속된 두 프레임 사이에서 변하지 않은 부분은 그대로 두고, 움직임이 발생한 픽셀의 변화량만 기록한다.
- 영상은 인코딩 방식에 따라 고정 비트율(CBR) 또는 가변 비트율(VBR)로 압축되어 다양한 화질(해상도)의 버전으로 생성된다.

### 2. HTTP 스트리밍과 DASH (동적 적응형 스트리밍)

과거의 단순한 HTTP 스트리밍은 서버가 정해진 화질의 영상 파일을 일방적으로 전송했다. 사용자의 인터넷 속도가 느려지면 영상이 멈추고 버퍼링이 걸리는 문제가 빈번했다. 이를 극복하기 위해 등장한 현대의 표준 기술이 DASH (Dynamic Adaptive Streaming over HTTP)이다.

**DASH의 동작 원리:**

1. **서버 측 준비:** 서버는 원본 비디오를 여러 화질(예: 4K, 1080p, 720p, 480p)로 인코딩하고, 각각을 수 초(예: 2~10초) 길이의 작은 파일 조각(Chunk)들로 나눈다.
2. **매니페스트 파일 (Manifest File):** 서버는 각 화질별 조각들의 URL 주소와 비트율 정보를 담은 '매니페스트 파일'을 클라이언트에게 먼저 제공한다.
3. **클라이언트 주도권 (Client-driven):** 비디오 플레이어(클라이언트)가 주도권을 쥐고 매 순간 최적의 화질을 **스스로 결정**한다.
- 클라이언트는 현재 자신의 네트워크 대역폭(다운로드 속도)과 버퍼에 남은 영상의 양을 실시간으로 측정한다.
- 네트워크 상태가 좋으면 고화질 조각을 HTTP GET으로 요청하고, 상태가 나빠지면 끊김을 막기 위해 즉시 저화질 조각을 요청한다.

### 3. 단일 메가 서버의 한계와 CDN의 등장

유튜브 규모의 서비스가 전 세계 사용자에게 DASH 비디오 조각을 하나의 거대한 중앙 서버(Mega-server)에서 전송한다고 가정해 보자.

- **단일 고장점:** 서버 하나가 죽으면 전 세계 서비스가 마비된다.
- **네트워크 혼잡:** 서버 주변의 네트워크 링크에 트래픽이 집중되어 심각한 병목 현상이 발생한다.
- **물리적 지연:** 한국에 있는 사용자가 미국 서버에서 영상을 받을 경우, 수많은 라우터를 거쳐야 하므로 지연 시간과 데이터 손실 확률이 극도로 높아진다.

이를 해결하기 위해 등장한 것이 CDN (Content Distribution Network, 콘텐츠 분배 네트워크)이다. CDN은 전 세계 곳곳의 지리적으로 분산된 수많은 서버에 비디오 복사본을 저장(캐싱)해 두고, 사용자와 가장 가까운 서버에서 데이터를 전송하게 하는 분산 인프라이다.

**CDN 서버 배치 철학:**

- **Enter Deep (깊은 침투):** 아카마이(Akamai) 같은 회사가 주로 사용하는 방식이다. 서버 클러스터를 전 세계 수많은 로컬 ISP(KT, SKT 등)의 접근 네트워크 내부 깊숙이 구축한다. 사용자와 물리적으로 가장 가까워 지연과 패킷 손실이 최소화되지만, 유지보수해야 할 서버 위치가 너무 많아 관리가 어렵다.
- **Bring Home (가까이 가져오기):** 더 적은 수의 핵심 지점(IXP, 인터넷 교환 노드)에 거대한 서버 클러스터를 구축한다. 'Enter Deep'보다는 사용자와의 거리가 멀 수 있지만, 구축 비용이 저렴하고 관리가 용이하다.

### 4. CDN의 동작 원리: DNS 리다이렉션 (DNS Redirection)

클라이언트가 비디오를 요청했을 때, 전 세계 수천 개의 CDN 서버 중 **가장 최적의(가까운) 서버를 어떻게 찾아 연결해 줄까?** 이때 2.4절에서 배운 **DNS**가 핵심적인 역할을 수행한다.

1. 사용자가 웹페이지에서 영화 링크(`http://video.netcinema.com/movie.mp4`)를 클릭한다.
2. 사용자의 로컬 DNS 서버가 `video.netcinema.com`의 IP 주소를 찾기 위해 DNS 질의를 시작한다.
3. `netcinema.com`의 책임 DNS 서버는 영상이 담긴 실제 IP 주소를 반환하는 대신, CDN 제공자(예: KingCDN)의 도메인 이름(예: `a1105.kingcdn.com`)으로 **CNAME(별명) 레코드**를 반환하여 쿼리를 **리다이렉션(방향 전환)** 시킨다.
4. 로컬 DNS 서버는 이제 CDN의 전용 DNS 서버에 질의를 보낸다.
5. **CDN DNS 서버의 똑똑한 결정:** 이 서버는 쿼리를 보낸 로컬 DNS 서버의 IP 주소를 확인하여 사용자의 지리적 위치를 파악하고, 실시간 네트워크 혼잡도를 고려하여 가장 빠르고 안전하게 영상을 보내줄 수 있는 최적의 **CDN 엣지 서버(Edge Server)의 IP 주소**를 최종적으로 반환한다.
6. 클라이언트는 이 IP 주소로 직접 TCP 연결을 맺고 DASH 방식으로 비디오 조각들을 요청하여 재생을 시작한다.

## 2.7 소켓 프로그래밍: 네트워크 애플리케이션 생성

### 1. UDP 소켓 프로그래밍 (편지 보내기 모델)

UDP는 **비연결형(Connectionless)** 프로토콜이다. 클라이언트와 서버가 사전에 통신 준비를 하는 과정(핸드셰이킹)이 없다.

- **동작 방식:** 우체통에 편지를 넣는 것과 완벽히 동일하다.
- **데이터 전송:** 클라이언트가 데이터를 보낼 때마다 데이터 덩어리(패킷)에 **목적지의 IP 주소와 포트(Port) 번호를 명시적으로 하나하나 붙여서** 네트워크(소켓)로 밀어 넣는다.
- **데이터 수신:** 서버는 소켓을 열어두고 기다리다가 패킷이 도착하면 데이터를 읽는다. 이때, 응답을 보내기 위해 **도착한 패킷에서 송신자(클라이언트)의 IP 주소와 포트 번호를 추출**하여 그 주소로 다시 답장을 보낸다.

**[동작 흐름 요약]**

1. **서버:** 특정 포트 번호를 열고(Bind) 무작정 수신 대기(`recvfrom`).
2. **클라이언트:** 데이터에 서버 주소를 붙여서 전송(`sendto`).
3. **서버:** 데이터를 받고, 봉투에 적힌 클라이언트 주소로 결과물 전송(`sendto`).

### 2. TCP 소켓 프로그래밍 (전화 걸기 모델)

TCP는 **연결지향형(Connection-oriented)** 프로토콜이다. 데이터를 주고받기 전에 반드시 둘 사이에 전용 파이프(TCP 연결)를 뚫어야 한다.

이 과정에서 교재가 가장 강조하는 핵심 개념은 서버 측에 **두 종류의 소켓**이 존재한다는 점이다.

### A. 환영 소켓 (Welcoming Socket)

- 서버가 켜지면 가장 먼저 생성되는 소켓이다.
- 이 소켓은 데이터를 주고받지 않는다. 오직 클라이언트들의 **'연결 요청(전화)'을 기다리고 수락하는 안내 데스크 역할**만 한다. (`listen`, `accept`)

### B. 연결 소켓 (Connection Socket)

- 클라이언트가 '환영 소켓'으로 연결을 요청(`connect`)하여 TCP 3-way handshake가 완료되면, 서버는 **해당 클라이언트와만 데이터를 전담해서 주고받을 새로운 소켓을 즉시 생성**한다. 이것이 '연결 소켓'이다.
- 안내 데스크(환영 소켓)에서 손님을 맞이한 뒤, 전담 상담원(연결 소켓)을 배정해 주는 방으로 안내하는 것과 같다. 환영 소켓은 다시 다른 클라이언트의 연결을 기다리러 돌아간다.

**[동작 흐름 요약]**

1. **서버:** 환영 소켓 생성 후 대기(`listen`).
2. **클라이언트:** 서버의 IP와 포트로 연결 요청(`connect`).
3. **서버:** 요청 수락 후, **새로운 연결 소켓 생성**(`accept`).
4. **데이터 송수신:** 이제 파이프가 연결되었으므로, 주소표를 매번 붙일 필요 없이 스트림(Stream) 형태로 데이터를 쏟아낸다(`send`, `recv`).
5. **종료:** 통신이 끝나면 연결 소켓을 닫는다(`close`).

### 💡 실무 개발 관점에서의 소켓 제어

소켓 API는 기본적으로 '블로킹(Blocking)' 방식으로 동작한다. 즉, 서버나 클라이언트가 데이터를 수신하는 `recv()` 함수를 호출하면, 데이터가 네트워크를 통해 들어올 때까지 프로그램이 그 줄에서 영원히 멈춰서 기다리게 된다.

실제 무인기나 드론 GCS(지상제어장치)와 같은 실시간 통신 애플리케이션을 개발할 때, 통신 모뎀에서 빈 데이터(Empty data)를 보내주기를 기다리지 않는다. 대신 PC 애플리케이션 내부의 독자적인 타이머를 구동하여, 소켓에서 1초 동안 아무런 수신 데이터가 없으면 즉시 '통신 단절(Disconnected)' 상태로 판별하고 화면에 표시하는 등의 앱 레벨 예외 처리 로직을 이 소켓 API 위에 덧붙여 구현하게 된다.
