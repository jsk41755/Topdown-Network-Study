<summary>재귀 질의 vs 반복 질의 장단점</summary>

### 1. 재귀 질의 (Recursive Query)

클라이언트가 DNS 서버에게 요청하면, DNS 서버가 끝까지 대신 조회해서 최종 결과를 돌려준다.

#### 장점

- 클라이언트는 한 번만 요청하면 된다.
- DNS 조회 로직을 클라이언트가 몰라도 된다.

#### 단점

- DNS 서버 부담이 크다.
- 공격 대상이 되기 쉽다.
- 재귀 DNS 서버가 장애가 나면 사용자가 DNS 조회를 못 한다.

### 2. 반복 질의 (Iterative Query)

DNS 서버가 최종 답을 모르면, 다음에 물어볼 DNS 서버 주소를 알려준다.

예시:

```text
로컬 DNS → Root DNS
Root DNS → ".com DNS에게 물어봐"

로컬 DNS → .com DNS
.com DNS → "google.com DNS에게 물어봐"

로컬 DNS → google.com DNS
google.com DNS → "IP는 x.x.x.x"
```

#### 장점

- 각 DNS 서버의 부담이 적다.
- Root/TLD DNS 서버가 모든 질의를 끝까지 처리하지 않아도 된다.

#### 단점

- 요청자가 여러 번 질의해야 한다.
- 구현이 재귀 질의보다 복잡하다.
- 최종 IP를 얻기까지 시간이 걸릴 수 있다.

### 실제 DNS에서의 동작

```text
내 PC → 로컬 DNS 서버 : 재귀 질의
로컬 DNS 서버 → Root/TLD/Authoritative DNS : 반복 질의
```

로컬 DNS 서버가 응답에 대한 책임을 가지며 캐싱 역할도 수행하므로, 모든 요청이 Root/TLD DNS 서버까지 가지 않게 된다.

<details>
<summary>DNS 캐싱 중 IP가 변경되면?</summary>

**네, TTL이 2일인 상태에서 IP를 변경하면 이미 기존 값을 캐시한 DNS Resolver는 남은 TTL 동안 기존 IP를 응답할 수 있다.**

그래서 DNS 변경 전에는 TTL을 미리 낮추고, 변경 후에는 기존 IP 서버를 TTL 이상 유지하는 방식으로 운영한다.

</details>

<details>
<summary>DNS Host 이름 별칭(CNAME)을 사용하는 이유</summary>

### 1. 사람이 기억하기 쉬운 이름 사용

사용자가 이해하기 쉬운 도메인 이름을 사용할 수 있다.

### 2. 실제 서버가 바뀌어도 사용자 주소는 유지됨 (중요)

기존:

```text
www.example.com → old-server.example.com
```

변경 후:

```text
www.example.com → new-server.example.com
```

DNS 설정만 변경하면 되므로 사용자 URL은 그대로 유지된다.

### 3. 서비스 이름과 서버 이름 분리

```text
www.example.com  → web-server-1.example.com
mail.example.com → mail-server-1.example.com
api.example.com  → api-server-1.example.com
```

서비스 관점의 이름과 실제 서버 이름을 분리할 수 있다.

### 4. 여러 별칭이 하나의 서버를 가리킬 수 있음

```text
www.example.com      CNAME d123.cloudfront.net
static.example.com   CNAME d123.cloudfront.net
assets.example.com   CNAME d123.cloudfront.net
```

실제 운영에서는 CDN, 로드밸런서, SaaS 서비스 연결 시 자주 사용된다.

</details>
