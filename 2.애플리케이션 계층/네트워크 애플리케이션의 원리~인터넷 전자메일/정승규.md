## 2.1 네트워크 애플리케이션의 원리

네트워크 애플리케이션을 개발한다는 것은 네트워크 코어(라우터, 링크 스위치 등)의 소프트웨어를 작성하는 것이 아니라, 서로 다른 종단 시스템(End system)에 존재하는 프로그램들이 네트워크를 통해 통신하도록 소프트웨어를 작성하는 것을 의미한다.

이러한 애플리케이션을 설계할 때 개발자가 고려해야 하는 핵심 원리와 구조는 다음과 같이 5가지 파트로 나뉜다.

### 1. 네트워크 애플리케이션 아키텍처

애플리케이션 개발자는 프로그램을 작성하기 전에 애플리케이션의 구조를 결정해야 한다. 교재에서는 주로 두 가지 구조를 다룬다.

- **클라이언트-서버 (Client-Server) 아키텍처:**
    - **특징:** 항상 켜져 있는 호스트인 '서버'가 여러 다른 호스트인 '클라이언트'의 요청을 받아 서비스한다.
    - **조건:** 서버는 고정된, 잘 알려진 IP 주소를 가져야 한다. 반면 클라이언트는 간헐적으로 연결될 수 있고 동적인 IP 주소를 가질 수 있으며, 클라이언트끼리는 서로 직접 통신하지 않는다. (예: 웹, 이메일)
    - **확장성:** 수많은 클라이언트의 요청을 감당하기 위해 단일 서버가 아닌 강력한 데이터 센터를 구축하여 스케일링(확장)한다.
- **P2P (Peer-to-Peer) 아키텍처:**
    - **특징:** 항상 켜져 있는 중앙 서버 인프라에 최소한으로 의존하거나 아예 의존하지 않는다. 대신 간헐적으로 연결되는 임의의 호스트들(피어, Peer) 쌍이 서로 직접 통신한다.
    - **장점 (자가 확장성):** 피어들이 파일을 요구함과 동시에 다른 피어들에게 파일을 분배(서비스)하기 때문에, 시스템에 참여하는 피어가 늘어날수록 시스템의 전체 서비스 능력도 함께 증가한다. (예: 비트토렌트)
    - **단점:** 고도의 분산 구조이므로 관리, 보안, 성능 보장이 클라이언트-서버 방식보다 훨씬 어렵다.

### 2. 프로세스 간 통신 (Processes Communicating)

네트워크에서 실제로 통신하는 주체는 기기(호스트) 자체가 아니라, 기기 안에서 실행되고 있는 '프로세스(Process)'이다.

- **클라이언트와 서버 프로세스:** 통신하는 두 프로세스 중, 통신 세션을 먼저 초기화(시작)하는 프로세스를 '클라이언트'라고 부르고, 접속을 기다리는 프로세스를 '서버'라고 부른다. (P2P 구조에서도 특정 파일을 요청하는 세션 내에서는 클라이언트와 서버의 역할이 명확히 나뉜다.)
- **소켓 (Socket):** 프로세스가 네트워크로 메시지를 보내거나 네트워크에서 메시지를 받기 위해 반드시 통과해야 하는 소프트웨어 인터페이스이다. 교재에서는 이를 '문(Door)'에 비유(교재 내 비유)한다. 애플리케이션 개발자는 이 소켓의 애플리케이션 계층 쪽(문 안쪽)에 대한 모든 통제권을 갖지만, 트랜스포트 계층 쪽(문 바깥쪽)에 대해서는 프로토콜 선택 등 극히 제한적인 통제권만 갖는다.
- **프로세스 주소 지정 (Addressing):** 한 프로세스가 다른 호스트의 특정 프로세스로 메시지를 보내려면 다음 두 가지 정보가 반드시 필요하다.
    1. 목적지 호스트를 식별하기 위한 **IP 주소 (IP Address)**
    2. 목적지 호스트 내에서 데이터를 받을 특정 프로세스(소켓)를 식별하기 위한 **포트 번호 (Port Number)** (예: 웹 서버는 80번 포트 사용)

### 3. 애플리케이션이 요구하는 트랜스포트 서비스

애플리케이션은 기능과 목적에 따라 하위 트랜스포트 계층에 다음과 같은 4가지 차원의 서비스를 요구한다.

- **신뢰적 데이터 전송 (Reliable Data Transfer):** 전송된 데이터가 손실 없이 100% 온전하게 도착하는 것을 보장하는 기능이다. 이메일, 웹 문서 전송 등은 데이터 유실을 전혀 허용하지 않지만, 실시간 오디오/비디오 스트리밍은 일부 패킷 손실을 허용(손실 톨러런트)한다.
- **처리율 (Throughput):** 애플리케이션이 제 기능을 하기 위해 보장받아야 하는 최소한의 전송 속도이다. 인터넷 전화처럼 특정 처리율이 반드시 보장되어야 하는 '대역폭 민감 애플리케이션'과, 주어진 만큼의 대역폭만 사용하는 '탄력적 애플리케이션(예: 파일 전송)'으로 나뉜다.
- **시간 (Timing):** 송신자가 패킷을 보낸 후 수신자에게 도착하기까지의 지연 시간에 대한 엄격한 제한을 의미한다. (예: 인터넷 전화, 멀티플레이어 게임 등에서 중요)
- **보안 (Security):** 기밀성(암호화), 데이터 무결성 보장 등의 서비스이다.

### 4. 인터넷이 제공하는 트랜스포트 서비스

인터넷은 애플리케이션에게 크게 TCP와 UDP라는 두 가지 트랜스포트 프로토콜을 제공한다.

- **TCP (Transmission Control Protocol):**
    - **연결 지향형 서비스:** 데이터를 보내기 전 클라이언트와 서버 사이에 사전 연결 설정(핸드셰이킹)을 수행한다.
    - **신뢰적 데이터 전송:** 송신한 모든 데이터가 순서대로, 오류나 손실 없이 수신될 것을 보장한다.
    - **혼잡 제어:** 네트워크가 혼잡해지면 송신 프로세스의 전송 속도를 스스로 낮추어 네트워크 전체의 마비를 방지한다.
- **UDP (User Datagram Protocol):**
    - 사전 연결 설정이 없는(비연결형) 매우 가벼운 프로토콜이다.
    - 신뢰성을 보장하지 않으므로 데이터가 손실되거나 순서가 뒤바뀔 수 있다.
    - 혼잡 제어 기능이 없으므로, 송신 프로세스는 원하는 속도로 데이터를 펌프질하듯 보낼 수 있다.
- **※ 교재의 핵심 지적:** 인터넷의 기본 모델(TCP, UDP 모두)은 애플리케이션이 요구하는 4가지 서비스 중 **'처리율'과 '시간(지연)'에 대한 어떠한 보장도 해주지 않는다.** (최선형 서비스, Best-effort).

### 5. 애플리케이션 계층 프로토콜

네트워크 애플리케이션과 '애플리케이션 계층 프로토콜'은 같은 것이 아니다. 웹 브라우저는 애플리케이션 그 자체이고, HTTP는 그 웹 브라우저가 통신하기 위해 사용하는 프로토콜(규칙)일 뿐이다.
애플리케이션 계층 프로토콜은 통신하는 프로세스 간에 다음 4가지를 명확히 정의한다.

1. 교환되는 메시지의 **유형** (예: 요청 메시지인지, 응답 메시지인지)
2. 여러 메시지 타입의 **문법(Syntax)** (메시지 안의 어떤 필드들이 어떻게 구별되는가)
3. 필드에 있는 정보의 **의미(Semantics)** (그 필드의 데이터를 어떻게 해석할 것인가)
4. 프로세스가 언제, 어떻게 메시지를 전송하고 응답하는지에 대한 **규칙**

## 2.2 웹과 HTTP

### 1. 웹의 기초 용어와 HTTP 개요

- **웹 페이지(Web Page)와 객체(Object):** 웹 페이지는 여러 객체들로 구성된다. 객체는 HTML 파일, JPEG 이미지, GIF 그래픽, 오디오 파일 등 단일 URL로 지정할 수 있는 하나의 파일이다. 대부분의 웹 페이지는 하나의 기본 HTML 파일과 이 파일이 참조하는 여러 객체들로 이루어진다.
- **URL(Uniform Resource Locator):** 객체의 주소를 나타내며, 호스트 이름(Host name)과 **경로 이름(Path name)** 두 부분으로 구성된다.
- **HTTP(HyperText Transfer Protocol):** 웹의 애플리케이션 계층 프로토콜이다. 클라이언트 프로그램과 서버 프로그램이 서로 다른 종단 시스템에서 실행되며, HTTP 메시지를 교환하여 통신한다.
- **TCP의 사용:** HTTP는 하위 트랜스포트 프로토콜로 TCP를 사용한다. HTTP 클라이언트는 먼저 서버와 TCP 연결을 설정(소켓 생성)한 뒤, 이 연결을 통해 무결하게 메시지를 전송한다.
- **비상태 프로토콜(Stateless Protocol):** HTTP 서버는 클라이언트에 대한 어떠한 상태 정보도 유지하지 않는다. 동일한 클라이언트가 몇 초 만에 같은 객체를 다시 요청하더라도, 서버는 이전에 보냈다는 사실을 기억하지 않고 객체를 새로 전송한다.

### 2. 비지속 연결과 지속 연결 (Non-persistent and Persistent Connections)

HTTP는 요청과 응답 쌍이 전달되는 방식에 따라 두 가지 연결 유형으로 나뉜다. 이를 이해하기 위해 교재에서는 RTT(Round-Trip Time)를 정의한다. RTT는 작은 패킷이 클라이언트로부터 서버까지 갔다가 다시 클라이언트로 돌아오는 데 걸리는 시간이다.

**A. 비지속 연결 (Non-persistent Connection)**

- **동작 방식:** 각 요청/응답 쌍이 별도의 독립적인 TCP 연결을 통해 전송된다. 즉, 하나의 객체를 전송할 때마다 TCP 연결이 새로 맺어지고 전송이 끝나면 닫힌다.
- **지연 시간 계산:** 하나의 객체를 가져오는 데 드는 총 시간은 2 * RTT + 서버의 파일 전송 시간이다.
    1. 클라이언트와 서버 간 TCP 연결을 설정하는 데 1 RTT가 소요된다.
    2. 클라이언트가 HTTP 요청 메시지를 보내고, 서버가 응답 메시지를 돌려받는 데 1 RTT가 소요된다.
- **단점:** 각 객체마다 연결 설정 오버헤드가 발생하며, 수많은 객체를 포함한 웹 페이지를 불러올 때 라우터와 호스트 자원에 큰 부담을 준다.

**B. 지속 연결 (Persistent Connection)**

- **동작 방식:** 서버는 응답을 보낸 후에도 TCP 연결을 그대로 유지한다. 따라서 같은 클라이언트와 서버 간의 후속 요청 및 응답은 동일한 연결을 통해 전송될 수 있다.
- **특징:** 새로운 객체를 요청할 때 TCP 연결 설정을 위한 1 RTT를 절약할 수 있다. 특히 파이프라이닝(Pipelining)을 사용하면 클라이언트는 이전 요청에 대한 응답을 기다리지 않고 연속해서 객체를 요청할 수 있어 지연 시간이 더욱 줄어든다.

### 3. HTTP 메시지 포맷

HTTP 메시지는 텍스트(ASCII) 형태로 작성되며, 요청(Request) 메시지와 응답(Response) 메시지로 구별된다.

**A. HTTP 요청 메시지 (Request Message)**

1. **요청 라인 (Request Line):** 메시지의 첫 줄로, **메서드 필드**, **URL 필드**, **HTTP 버전 필드**로 구성된다.
    - *메서드 종류:* `GET` (객체 요청), `POST` (폼 입력 데이터 등을 서버 몸체에 실어 전송), `HEAD` (객체 본문 없이 헤더만 요청), `PUT` (서버에 파일 업로드), `DELETE` (서버의 파일 삭제)
2. **헤더 라인 (Header Lines):** `Host` (목적지 호스트 주소), `User-Agent` (브라우저 타입), `Connection` (연결 유지 여부), `Accept-language` (선호 언어) 등의 추가 정보를 서버에 제공한다.
3. **공백 라인 (Blank Line):** 헤더의 끝을 나타내는 캐리지 리턴과 라인 피드(CRLF)이다.
4. **개체 몸체 (Entity Body):** `POST` 메서드를 사용할 때 사용자가 입력한 실제 데이터가 들어가는 부분이다. `GET` 메서드일 때는 비어 있다.

**B. HTTP 응답 메시지 (Response Message)**

1. **상태 라인 (Status Line):** **HTTP 버전**, **상태 코드**, **상태 문구**로 구성된다.
2. **헤더 라인 (Header Lines):** `Connection`, `Date` (응답 생성 시간), `Server` (웹 서버 소프트웨어 정보), `Last-Modified` (객체의 최종 수정 시간), `Content-Length` (바이트 수), `Content-Type` (객체의 타입) 등이 포함된다.
3. **공백 라인 (Blank Line)**
4. **개체 몸체 (Entity Body):** 클라이언트가 요청한 실제 파일(데이터) 내용이 들어간다.

> **※ 대표적인 상태 코드 (Status Codes)**
> 
> - `200 OK`: 요청이 성공했고, 정보가 응답에 포함되어 있음.
> - `301 Moved Permanently`: 요청한 객체가 영구적으로 이동함 (새 URL은 헤더에 명시).
> - `400 Bad Request`: 서버가 요청을 이해할 수 없음 (일반적인 문법 오류).
> - `404 Not Found`: 요청한 문서가 서버에 존재하지 않음.
> - `505 HTTP Version Not Supported`: 요청된 HTTP 프로토콜 버전을 서버가 지원하지 않음.

### 4. 사용자-서버 상호작용: 쿠키 (Cookies)

HTTP는 비상태 프로토콜이지만, 웹사이트가 사용자를 식별하고 상태를 유지(예: 쇼핑 카트, 로그인 상태 관리)하기 위해 **쿠키(Cookie)** 기술을 사용한다. 쿠키 기술은 다음 4가지 요소로 구성된다.

1. HTTP 응답 메시지에 포함되는 쿠키 헤더 라인 (`Set-Cookie:`)
2. HTTP 요청 메시지에 포함되는 쿠키 헤더 라인 (`Cookie:`)
3. 사용자의 종단 시스템에 저장되고 브라우저에 의해 관리되는 쿠키 파일
4. 웹사이트 백엔드 데이터베이스의 사용자 상태 정보

서버는 클라이언트가 처음 접속했을 때 고유한 식별 번호(ID)를 생성하여 백엔드 데이터베이스에 저장하고, 응답 헤더의 `Set-Cookie:`를 통해 클라이언트에게 전달한다. 이후 브라우저는 해당 사이트에 요청을 보낼 때마다 자동으로 헤더에 `Cookie: [ID]`를 포함하여 전송하므로, 서버는 클라이언트를 식별할 수 있게 된다.

### 5. 웹 캐싱 (Web Caching)

웹 캐시(Web Cache)는 기원 서버(Origin server)를 대신하여 HTTP 요구를 충족시키는 네트워크 개체로, 흔히 프록시 서버(Proxy Server)라고도 부른다. 자체 디스크 저장소를 갖추고 있어 최근에 호출된 객체의 사본을 보관한다.

- **동작 원리:** 클라이언트의 모든 HTTP 요청은 먼저 웹 캐시로 전송된다. 캐시에 요청한 객체의 사본이 있으면 즉시 클라이언트에게 반환하고, 없으면 기원 서버에 객체를 요청하여 가져온 뒤 사본을 저장하고 클라이언트에게 전달한다.
- **이점:** 클라이언트의 응답 시간을 크게 줄일 수 있으며, 기관의 인터넷 접속 링크 트래픽을 감소시켜 전체적인 네트워크 성능을 향상시킨다.
- **조건부 GET (Conditional GET):** 캐시에 저장된 객체의 사본이 최신 상태인지 확인하기 위한 메커니즘이다. 캐시는 기원 서버에 요청을 보낼 때 헤더 라인에 `If-Modified-Since: [보유한 사본의 최종 수정일]`을 포함하여 보낸다. 만약 객체가 변경되지 않았다면, 서버는 본문을 제외한 채 **`304 Not Modified`** 상태 코드만 담은 응답을 보내 대역폭 낭비를 막는다.

## 2.3 인터넷 전자메일

### 1. 인터넷 메일 시스템의 3가지 핵심 요소

인터넷 메일 시스템은 크게 세 가지 주요 구성 요소로 이루어진다.

- **사용자 에이전트 (User Agent):**
    - 사용자가 메일을 읽고, 작성하고, 회신하고, 전달할 수 있게 해주는 소프트웨어다. (예: 아웃룩, 애플 메일 등)
    - 작성된 메일은 사용자 에이전트에서 사용자의 '메일 서버'로 보내져 그곳의 큐(Queue)에 저장된다.
- **메일 서버 (Mail Server):**
    - 인터넷 이메일 인프라의 중심(Core)이다.
    - **메일박스(Mailbox):** 각 사용자는 메일 서버 안에 자신만의 메일박스를 가지며, 수신된 메일들이 이곳에 보관된다.
    - **메시지 큐(Message Queue):** 송신자가 보낼 메일들이 목적지 서버로 전송되기 전 임시로 대기하는 공간이다. 만약 목적지 서버가 고장 났다면 일정 시간 간격으로 재전송을 시도하고, 며칠이 지나도 실패하면 송신자에게 실패 메시지를 반환한다.
- **SMTP (Simple Mail Transfer Protocol):**
    - 인터넷 이메일의 핵심 애플리케이션 계층 프로토콜이다.
    - 송신자의 메일 서버에서 수신자의 메일 서버로 메일을 전송할 때 사용한다.

### 2. SMTP의 동작 방식과 특징

- **TCP 사용:** SMTP는 데이터의 손실 없는 신뢰성 있는 전송을 위해 하위 트랜스포트 프로토콜로 TCP(포트 번호 25)를 사용한다.
- **직접 연결:** 중간에 다른 메일 서버를 거치지 않고, 송신 서버와 수신 서버가 직접 TCP 연결을 맺고 통신한다.
- **푸시(Push) 프로토콜:** 송신 메일 서버가 수신 메일 서버로 파일을 '밀어 넣는' 방식이다.
- **7비트 ASCII 제한:** SMTP는 메시지의 헤더와 몸체 모두가 반드시 7비트 ASCII 형식으로 인코딩되어야 한다는 오래된 제약 조건을 가지고 있다. (텍스트가 아닌 멀티미디어 데이터는 전송 전 ASCII 형식으로 변환되어야 한다.)
- **명령어와 응답:** 클라이언트(송신 서버)와 서버(수신 서버)는 사람이 읽을 수 있는 텍스트 기반의 명령어(`HELO`, `MAIL FROM`, `RCPT TO`, `DATA`, `QUIT`)와 상태 코드 및 문구를 주고받으며 통신한다.

### 3. SMTP와 HTTP의 비교

교재에서는 두 프로토콜을 다음과 같이 명확히 대조된다.

| **구분** | **HTTP (웹)** | **SMTP (이메일)** |
| --- | --- | --- |
| **통신 방식** | **풀(Pull) 프로토콜:** 사용자가 원할 때 서버로부터 문서를 가져온다. | **푸시(Push) 프로토콜:** 송신 서버가 수신 서버로 데이터를 밀어 넣는다. |
| **데이터 형식** | 데이터 형식에 제한이 없다. | 데이터를 반드시 **7비트 ASCII**로 변환해야 한다. |
| **문서 처리** | 웹 페이지의 각 객체(이미지, HTML 등)를 개별적인 HTTP 응답 메시지로 캡슐화한다. | 메일 안의 모든 객체(텍스트, 첨부파일 등)를 **하나의 큰 메시지** 안에 모두 담아서 보낸다. |

### 4. 메일 접속 프로토콜 (Mail Access Protocols)

**왜 별도의 접속 프로토콜이 필요한가?**
Alice가 Bob에게 메일을 보낼 때, Alice의 메일 서버는 Bob의 메일 서버로 메일을 푸시(SMTP)한다. 하지만 Bob이 자신의 PC(사용자 에이전트)에서 메일을 확인하려면, 자신의 메일 서버로부터 메일을 **가져와야(Pull)** 한다. SMTP는 푸시 프로토콜이므로 이 단계에서는 사용할 수 없으며, 다음과 같은 별도의 메일 접속 프로토콜이 필요하다.

**A. POP3 (Post Office Protocol - Version 3)**

- 매우 단순한 메일 접속 프로토콜이다.
- 사용자 에이전트가 서버에 접속하여 메일을 다운로드하는 역할만 한다.
- **"다운로드 후 삭제"** 모드에서는 사용자가 한 기기에서 메일을 읽으면 서버에서 지워지므로, 다른 기기에서 다시 읽을 수 없다. ("다운로드 후 유지" 모드로 설정할 수는 있다.)
- **상태 비유지:** 세션 간에 사용자의 상태(어떤 메일을 읽었는지, 삭제했는지 등)를 기록하지 않는다.

**B. IMAP (Internet Mail Access Protocol)**

- POP3보다 훨씬 복잡하고 강력한 프로토콜이다.
- 서버에 **폴더**를 생성하고 메일을 분류하여 관리할 수 있게 해준다.
- 사용자 에이전트가 서버의 메일 일부(예: 헤더만)만 가져오는 기능을 지원하여, 대용량 첨부파일을 다운로드하기 전에 미리 확인할 수 있다.
- **상태 유지:** 서버가 폴더 계층 구조와 각 메시지의 상태(읽음/안 읽음 등)를 세션 간에 지속적으로 유지한다.

**C. 웹 기반 이메일 (HTTP)**

- 오늘날 Gmail, 네이버 메일 등에서 가장 많이 사용하는 방식이다.
- 사용자 에이전트가 웹 브라우저가 되며, 사용자와 사용자의 메일 서버 간의 통신(메일 보내기 및 가져오기 모두)은 **HTTP**를 통해 이루어진다.
- 하지만 **메일 서버와 메일 서버 사이의 통신은 여전히 SMTP**를 사용한다.
